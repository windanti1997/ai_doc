# 公众号文章写作 Skill

> 通用公众号技术长文写作流程，适用于任意主题的 3000-5000 字深度文章。

---

## 一、什么时候用

用户说以下任一内容时，调用本 Skill：
- "写一篇公众号文章关于..."
- "帮我写篇推文介绍..."
- "写篇长文分析一下..."
- "按照上次 Trellis 那篇的风格写..."
- 任何明确要求产出公众号/微信/长文形式文章的请求

---

## 二、7 步写作法（必按顺序执行）

```
需求拆解 → 深度调研 → 广度调研 → 竞品对比 → 结构设计 → 正文撰写 → 质量审查
```

每一步都用 TodoWrite 跟踪进度，不达标不进入下一步。

### Step 1：需求拆解

**做什么**：
1. 启用 anti-overfit-collaboration 协议
2. 判断用户需求类型（执行/判断/发散/批评/未澄清）
3. 用 TodoWrite 拆成 6-8 个任务，每个有明确产出

**检查**：
- [ ] 任务是否拆解到单步可执行？
- [ ] 是否有明确的完成标准？

### Step 2：深度调研（官方文档）

**做什么**：
1. 优先读官方文档：首页、核心概念、工作原理、真实场景
2. 用 WebFetch 精读 3-5 篇核心文档
3. 提取：定位隐喻、核心架构、关键术语、推荐场景、FAQ 顾虑

**检查**：
- [ ] 能用一句话说清产品是干什么的？
- [ ] 能说出 3 个核心设计？
- [ ] 能说出 5 个使用场景？

### Step 3：广度调研（外部社区）

**做什么**：
1. 多渠道搜索：Reddit / HN / X / 知乎 / 掘金 / 独立博客
2. 关键词组合：`[主题] review/experience/评测/对比/提效`
3. 用 WebFetch 精读 3-5 篇高质量内容
4. 提取素材：真实推荐语、使用案例、踩坑记录、数字对比

**检查**：
- [ ] 至少 3 个独立来源？
- [ ] 有正面也有负面评价？
- [ ] 有具体案例或数字？

### Step 4：竞品对比

**做什么**：
1. 识别 2-4 个主要竞品/同类方案
2. 每个竞品搞清楚：定位、解决的问题、优缺点、本质区别
3. 建立 5+ 维度的对比表
4. 每个竞品先肯定再指出局限，不踩一捧一

**检查**：
- [ ] 竞品分析公平吗？
- [ ] 对比维度有说服力吗？
- [ ] 读者能理解"为什么选这个"吗？

### Step 5：结构设计

**做什么**：按下面的「爆款结构模板」设计大纲，确保情绪递进。

**检查**：
- [ ] 开头有钩子吗？
- [ ] 情绪是递进的吗？
- [ ] 有真实案例吗？
- [ ] 结尾有互动引导吗？

### Step 6：正文撰写

**做什么**：按结构写完整文章，3500-5000 字。

**写作原则**（必须遵守）：
1. 第一人称叙事，有代入感
2. 用具体场景，不用抽象概念
3. 先讲问题，再讲方案
4. 正反都讲，有 Good 有 Bad
5. 每 300-500 字一句金句
6. 小标题用判断句，不用名词短语

**检查**：
- [ ] 标题有吸引力吗？
- [ ] 开头 3 段能抓住人吗？
- [ ] 至少 3-5 句金句？
- [ ] 案例真实具体吗？
- [ ] 结尾有互动引导吗？
- [ ] 字数 3500-5000？

### Step 7：质量审查

**做什么**：调用 anti-overfit-reviewer 做最终检查。

**审查维度**：
- 有没有讨好用户、忽略事实？
- 有没有把推演说成事实？
- 有没有虚假精确数字？
- 交付物完整可用吗？
- 有没有关键假设没说明？

FAIL 就修改重来，不允许放水。

## 三、爆款结构模板（必用）

```
1. 【钩子】日常场景 → 制造冲突 → 引发共鸣
   读者熟悉的画面 → 点出痛点 → 一句话升级

2. 【痛点放大】把隐性痛点显性化
   拆解具体表现 → 对比句强化 → 说到读者心里

3. 【探索过程】我试过的方案，各有各的问题
   2-3 个同类方案 → 先肯定再讲局限 → 读者一起失望 → 产生期待

4. 【主角登场】答案揭晓，核心价值
   好的隐喻 → 分层讲架构（3 层最佳） → 每层讲清"解决了什么问题"

5. 【真实案例】Good Case / Bad Case
   ≥3 个 Good Case（不同场景） → ≥2 个 Bad Case（显得真实）
   每个案例：背景 → 怎么做 → 结果

6. 【场景指南】不同情况怎么用
   4-6 个典型场景 → 适合时机/价值/怎么做 → 加对比表

7. 【解除顾虑】FAQ
   提前想到读者会问的问题 → 直接回答不回避

8. 【金句收尾】升华主题 + 行动引导
   有力比喻 → 回到开头闭环 → 自然引导互动
```

## 四、Good Case（这样写是对的）

### ✅ Good Case 1：用具体场景开头

> 早上打开 Claude Code，说"帮我把用户邀请功能做了"。AI 噼里啪啦写了一堆文件，你一看，嗯，大体方向对。
> 下午换 Cursor 调 UI 细节，发现上午写的 API 字段名和前端对不上。
> 第二天开新会话，AI 问你："这个项目用什么 ORM？测试怎么跑？"

**为什么好**：
- 三个连续画面，读者立刻代入
- 都是真实日常，不是空泛的"上下文丢失"
- 情绪层层递进：还行 → 有点烦 → 无奈了

### ✅ Good Case 2：先讲问题再讲方案

**不说**："Trellis 有 spec 规范层，可以管理项目规范。"
**而是说**："以前你写了 1000 行 CLAUDE.md，AI 真的读完了吗？你不确定。Trellis 的 spec 层解决的就是这个问题——按需加载、自动注入、自更新。"

**为什么好**：
- 先唤醒痛点记忆
- 再说方案，读者立刻懂价值
- 不是推销功能，是解决问题

### ✅ Good Case 3：正反案例都有

3 个 Good Case（老项目接入 / 团队推广 / 修复反复 Bug）
+ 2 个 Bad Case（一开始写满 spec / 所有任务都套流程）

**为什么好**：
- 正面案例给信心
- 反面案例显真实
- 读者能避开坑，信任感更强

### ✅ Good Case 4：用比喻收尾

> 模型是发动机。
> 但只有发动机，没有方向盘、没有刹车、没有导航，车开不快，也开不远。
> Trellis 就是那套方向盘+刹车+导航。

**为什么好**：
- 比喻形象好记
- 回到开头的痛点，闭环
- 有余味，读者愿意转发

## 五、Bad Case（千万别这样写）

### ❌ Bad Case 1：空泛开头

> 人工智能正在改变软件开发的方式，AI 编程工具越来越普及，如何提高效率成为大家关注的话题...

**问题**：
- 全是正确的废话
- 读者看完第一段不知道你要说什么
- 没有冲突，没有钩子，直接划走

### ❌ Bad Case 2：一边倒吹捧

> Trellis 是最好用的 AI 编程工具，功能强大，体验完美，强烈推荐所有人使用！

**问题**：
- 没有缺点 = 不可信
- 像广告软文，不像真实分享
- 读者会本能抵触

### ❌ Bad Case 3：全是大道理没案例

> 规范很重要，记忆很重要，流程很重要，团队协作很重要...

**问题**：
- 读者看完不知道具体怎么做
- 没有场景，没有画面，没有代入感
- 看完就忘，不会转发

### ❌ Bad Case 4：AI 味重的模板化结构

> 1. 引言
> 2. 什么是 Trellis
> 3. Trellis 的功能
> 4. Trellis 的优势
> 5. 总结

**问题**：
- 教科书式结构，没人愿意读
- 没有情绪曲线，平铺直叙
- 像产品说明书，不像公众号文章

### ❌ Bad Case 5：虚假精确数字

> Trellis 可以提升 37.5% 的开发效率，减少 62% 的重复沟通。

**问题**：
- 没有来源的精确数字 = 编的
- 一眼假，读者立刻不信任
- 违反反过拟合原则

**正确姿势**：如果没有实测，就说"体感上明显提速"或者"我自己的感受是，第三周开始明显变快"，并标注是个人体验而非实测数据。

## 六、质量红线（违反就不能交付）

1. **没调研就写**：只靠内部知识，不查官方文档，不搜外部社区
2. **一边倒吹捧**：只讲优点，不讲缺点和局限
3. **空泛无物**：全是大道理，没有具体案例和场景
4. **AI 味重**：结构僵化，语言模板化，不像真人写的
5. **虚假精确**：把推演数字说成实测事实
6. **没有互动**：结尾干巴巴，不引导读者评论

# 公众号文章写作 Skill

> 通用公众号技术长文写作流程，适用于任意主题的 3000-5000 字深度文章。
> 核心原则：结构服务于主题，每篇必须有读者能带走的明确价值。

---

## 一、什么时候用

用户说以下任一内容时，调用本 Skill：
- "写一篇公众号文章关于..."
- "帮我写篇推文介绍..."
- "写篇长文分析一下..."
- "按照上次 XX 那篇的风格写..."
- 任何明确要求产出公众号/微信/长文形式文章的请求

---

## 二、7 步写作法（必按顺序执行）

```
需求拆解 → 深度调研 → 广度调研 → 竞品对比 → 结构设计 → 正文撰写 → 质量审查
```

每一步都用 TodoWrite 跟踪进度，不达标不进入下一步。

### Step 1：需求拆解

**做什么**：
1. 启用 anti-overfit-collaboration 协议
2. 判断用户需求类型（执行/判断/发散/批评/未澄清）
3. 用 TodoWrite 拆成 6-8 个任务，每个有明确产出
4. **明确读者价值**：这篇文章读者看完能带走什么？（方法/清单/工具/认知升级/避坑指南）

**检查**：
- [ ] 任务是否拆解到单步可执行？
- [ ] 是否有明确的完成标准？
- [ ] 读者价值想清楚了吗？（答不上来就别动笔）

### Step 2：深度调研（官方文档）

**做什么**：
1. 优先读官方文档：首页、核心概念、工作原理、真实场景
2. 用 WebFetch 精读 3-5 篇核心文档
3. 提取：定位隐喻、核心架构、关键术语、推荐场景、FAQ 顾虑
4. **收集可操作素材**：读者看完能直接用的方法、步骤、清单

**检查**：
- [ ] 能用一句话说清产品是干什么的？
- [ ] 能说出 3 个核心设计？
- [ ] 能说出 5 个使用场景？
- [ ] 有没有读者可以直接带走的方法论或工具？

### Step 3：广度调研（外部社区）

**做什么**：
1. 多渠道搜索：Reddit / HN / X / 知乎 / 掘金 / 独立博客
2. 关键词组合：`[主题] review/experience/评测/对比/提效`
3. 用 WebFetch 精读 3-5 篇高质量内容
4. 提取素材：真实推荐语、使用案例、踩坑记录、数字对比
5. **收集全新案例**：不要复用之前文章用过的案例和表达

**检查**：
- [ ] 至少 3 个独立来源？
- [ ] 有正面也有负面评价？
- [ ] 有具体案例或数字？
- [ ] 案例是全新的，没有在之前文章里用过？

### Step 4：竞品对比

**做什么**：
1. 识别 2-4 个主要竞品/同类方案
2. 每个竞品搞清楚：定位、解决的问题、优缺点、本质区别
3. 建立 5+ 维度的对比表
4. 每个竞品先肯定再指出局限，不踩一捧一
5. **对比服务于核心观点**：不是为了对比而对比，是为了说明"为什么这条路走不通，必须走主角这条路"

**检查**：
- [ ] 竞品分析公平吗？
- [ ] 对比维度有说服力吗？
- [ ] 读者能理解"为什么选这个"吗？
- [ ] 对比是在服务主题，还是在凑字数？

### Step 5：结构设计

**做什么**：
1. 先想清楚核心观点和读者价值
2. 根据主题**定制结构**，不硬套固定模板
3. 参考"爆款结构要素"，但灵活组合，每篇结构都应该不一样
4. 设计情绪递进曲线：共鸣 → 焦虑 → 期待 → 理解 → 收获 → 行动
5. **预留"读者价值"模块**：方法/清单/工具/自检表，必须有明确的一块是读者可以直接拿走用的

**结构设计铁则**：
- 没有哪一段是"因为模板里有所以必须有"的
- 每个模块都必须服务于核心观点
- 同一主题的不同角度，结构应该完全不同
- 读者能感觉到"这篇文章就是为这个问题量身定做的"

**检查**：
- [ ] 开头有钩子吗？
- [ ] 情绪是递进的吗？
- [ ] 有真实案例吗？
- [ ] 读者价值明确吗？（看完能带走什么）
- [ ] 结构是为主题定制的，还是在硬套模板？
- [ ] 结尾有互动引导吗？

### Step 6：正文撰写

**做什么**：按结构写完整文章，3500-5000 字。

**写作原则**（必须遵守）：
1. 第一人称叙事，有代入感
2. 用具体场景，不用抽象概念
3. 先讲问题，再讲方案
4. 正反都讲，有 Good 有 Bad
5. 每 300-500 字一句金句
6. 小标题用判断句，不用名词短语
7. **结构服务主题**：不为凑模板硬加段落，不需要的模块直接砍掉
8. **读者必须有收获**：每篇至少有一样读者可以直接带走用的东西（方法/清单/自检表/工具/框架）
9. **案例全新**：不复用之前文章的案例、故事、表达，每篇都用新素材

**检查**：
- [ ] 标题有吸引力吗？
- [ ] 开头 3 段能抓住人吗？
- [ ] 至少 3-5 句金句？
- [ ] 案例真实具体吗？
- [ ] 读者价值清晰可操作吗？
- [ ] 有没有模板化痕迹？（"做的事情很朴素"、"三个月"之类的固定表达）
- [ ] 结尾有互动引导吗？
- [ ] 字数 3500-5000？

### Step 7：质量审查

**做什么**：调用 anti-overfit-reviewer 做最终检查。

**审查维度**：
- 有没有讨好用户、忽略事实？
- 有没有把推演说成事实？
- 有没有虚假精确数字？
- 交付物完整可用吗？
- 有没有关键假设没说明？
- **有没有硬套模板的痕迹？**
- **读者价值明确吗？还是只提问题不给解法？**
- **案例和表达是不是全新的，有没有复用旧文？**

FAIL 就修改重来，不允许放水。

## 三、爆款结构要素（参考，不是必用）

以下是经过验证的有效结构模块，但**不是每篇都要全用上**。根据主题挑合适的，灵活组合。

```
【钩子模块】日常场景 → 制造冲突 → 引发共鸣
   读者熟悉的画面 → 点出痛点 → 一句话升级

【痛点放大模块】把隐性痛点显性化
   拆解具体表现 → 对比句强化 → 说到读者心里

【方案对比模块】我试过的方案，各有各的问题
   2-3 个同类方案 → 先肯定再讲局限 → 读者一起失望 → 产生期待
   （不是产品评测类文章，可以不要这模块）

【核心框架模块】答案揭晓，核心价值
   好的隐喻 → 分层讲架构 → 每层讲清"解决了什么问题"
   （方法论类文章用"步骤/原则/方法"代替"架构"）

【真实案例模块】Good Case / Bad Case
   ≥3 个 Good Case（不同场景） → ≥2 个 Bad Case（显得真实）
   每个案例：背景 → 怎么做 → 结果

【自检/工具模块】读者现在就能用
   自检清单 / 步骤法 / 工具表 / 模板
   （必须有！读者看完要能带走实实在在的东西）

【场景指南模块】不同情况怎么用
   4-6 个典型场景 → 适合时机/价值/怎么做 → 加对比表
   （应用型文章需要，纯认知型可以不要）

【解除顾虑模块】FAQ
   提前想到读者会问的问题 → 直接回答不回避
   （有争议或容易误解的主题需要）

【金句收尾模块】升华主题 + 行动引导
   有力比喻 → 回到开头闭环 → 自然引导互动
```

**怎么用**：
- 从上面挑 5-6 个模块，组合成你这篇的结构
- 顺序可以调，模块可以砍，一切为核心观点服务
- 同一主题的不同文章，结构应该不一样
- 如果某篇文章用了和上一篇一模一样的 8 段结构，要警惕——大概率是在硬套模板

## 四、Good Case（这样写是对的）

### ✅ Good Case 1：用具体场景开头

> 早上打开 Claude Code，说"帮我把用户邀请功能做了"。AI 噼里啪啦写了一堆文件，你一看，嗯，大体方向对。
> 下午换 Cursor 调 UI 细节，发现上午写的 API 字段名和前端对不上。
> 第二天开新会话，AI 问你："这个项目用什么 ORM？测试怎么跑？"

**为什么好**：
- 三个连续画面，读者立刻代入
- 都是真实日常，不是空泛的"上下文丢失"
- 情绪层层递进：还行 → 有点烦 → 无奈了

### ✅ Good Case 2：先讲问题再讲方案

**不说**："Trellis 有 spec 规范层，可以管理项目规范。"
**而是说**："以前你写了 1000 行 CLAUDE.md，AI 真的读完了吗？你不确定。Trellis 的 spec 层解决的就是这个问题——按需加载、自动注入、自更新。"

**为什么好**：
- 先唤醒痛点记忆
- 再说方案，读者立刻懂价值
- 不是推销功能，是解决问题

### ✅ Good Case 3：读者有实实在在的收获

一篇讲"过期记忆"的文章，不是只说"过期记忆很可怕"，而是给了：
- 10 题自检清单（读者马上能测）
- 3 步清理法（读者今天就能做）
- 4 种活记忆写法（读者以后记的时候就用上）

**为什么好**：
- 读者不是看完只记住"哦有这么个事"
- 而是看完就能动手解决自己的问题
- 有行动价值，读者会收藏会转发

### ✅ Good Case 4：结构为主题定制

讲"AI 记忆过期"的文章，用的是：
翻车现场 → 三类症状 → 自检清单 → 为什么会这样 → 3 步清理法 → 4 种活记忆写法 → 不同人群建议

而不是硬套"问题→竞品对比→三层架构→Good/Bad Case→场景→FAQ"的固定模板。

**为什么好**：
- 读者感觉"这篇就是为这个问题写的"
- 每一段都在推进主题，没有凑字数的模块
- 不会有"似曾相识"的模板感

### ✅ Good Case 5：用比喻收尾

> 模型是发动机。
> 但只有发动机，没有方向盘、没有刹车、没有导航，车开不快，也开不远。
> Trellis 就是那套方向盘+刹车+导航。

**为什么好**：
- 比喻形象好记
- 回到开头的痛点，闭环
- 有余味，读者愿意转发

## 五、Bad Case（千万别这样写）

### ❌ Bad Case 1：空泛开头

> 人工智能正在改变软件开发的方式，AI 编程工具越来越普及，如何提高效率成为大家关注的话题...

**问题**：
- 全是正确的废话
- 读者看完第一段不知道你要说什么
- 没有冲突，没有钩子，直接划走

### ❌ Bad Case 2：一边倒吹捧

> Trellis 是最好用的 AI 编程工具，功能强大，体验完美，强烈推荐所有人使用！

**问题**：
- 没有缺点 = 不可信
- 像广告软文，不像真实分享
- 读者会本能抵触

### ❌ Bad Case 3：全是大道理没案例

> 规范很重要，记忆很重要，流程很重要，团队协作很重要...

**问题**：
- 读者看完不知道具体怎么做
- 没有场景，没有画面，没有代入感
- 看完就忘，不会转发

### ❌ Bad Case 4：AI 味重的教科书结构

> 1. 引言
> 2. 什么是 Trellis
> 3. Trellis 的功能
> 4. Trellis 的优势
> 5. 总结

**问题**：
- 教科书式结构，没人愿意读
- 没有情绪曲线，平铺直叙
- 像产品说明书，不像公众号文章

### ❌ Bad Case 5：虚假精确数字

> Trellis 可以提升 37.5% 的开发效率，减少 62% 的重复沟通。

**问题**：
- 没有来源的精确数字 = 编的
- 一眼假，读者立刻不信任
- 违反反过拟合原则

**正确姿势**：如果没有实测，就说"体感上明显提速"或者"我自己的感受是，第三周开始明显变快"，并标注是个人体验而非实测数据。

### ❌ Bad Case 6：硬套模板，篇篇一样

每篇文章都是一模一样的 8 段：
1. 日常场景钩子
2. 痛点放大
3. 三种方案对比
4. 主角登场，三层架构
5. 3 Good + 2 Bad
6. 场景指南
7. FAQ
8. 金句收尾

连表达都固定："做的事情很朴素"、"三个月用下来"、"最后说句实在话"...

**问题**：
- 读者看两篇就腻了，似曾相识的感觉很不好
- 结构不为主题服务，是主题往结构里塞
- 像流水线生产的内容，没有灵魂

**正确姿势**：每篇根据主题定制结构。需要对比才对比，不需要就砍掉。需要 FAQ 才 FAQ，不需要就不要硬加。

### ❌ Bad Case 7：只提问题，不给解法

> AI 记忆过期很可怕，会误导你，会拖后腿，会让你越用越不顺手...
> （讲了一堆问题，然后就没了）

**问题**：
- 读者看完焦虑了，但不知道怎么办
- 只有认知冲击，没有行动价值
- 读者会觉得"说了一堆正确的废话"

**正确姿势**：每篇至少给一样读者能带走的东西——可以是方法、步骤、清单、自检表、工具、模板。哪怕是很小的一个东西，也比只提问题强。

### ❌ Bad Case 8：复用旧案例、旧表达

写第二篇文章的时候，还在用第一篇的案例、故事、甚至句子。
读者如果连着看两篇，立刻就能发现"怎么跟上次那个套路一样"。

**问题**：
- 偷懒，没有为每篇文章找新素材
- 读者会觉得你江郎才尽
- 信任度下降

**正确姿势**：每篇文章的案例、故事、主要表达都要是新的。哪怕是同一个主题的不同角度，也要找新的切入点、新的例子、新的说法。

## 六、质量红线（违反就不能交付）

1. **没调研就写**：只靠内部知识，不查官方文档，不搜外部社区
2. **一边倒吹捧**：只讲优点，不讲缺点和局限
3. **空泛无物**：全是大道理，没有具体案例和场景
4. **AI 味重**：结构僵化，语言模板化，不像真人写的
5. **虚假精确**：把推演数字说成实测事实
6. **没有互动**：结尾干巴巴，不引导读者评论
7. **硬套模板**：结构和表达明显复用前作，读者一眼就能看出来
8. **只有问题没有解法**：提了一堆问题，但读者看完不知道怎么办
9. **复用旧素材**：案例、故事、核心表达和之前文章重复

文章文件命名：`NN_主题_keyword.md`

统一放在项目对应目录下，确保路径清晰。

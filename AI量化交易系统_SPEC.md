## Problem Statement

用户是一名**不懂量化策略**的个人投资者，想用 AI 辅助做**加密货币现货**交易，但面临一连串真实困境：

- 不会写策略代码，也看不懂策略好坏，容易被"回测很美"的策略骗（过拟合）。
- 市面上号称"全闭环"的 AI 量化项目（如 QuantDinger）star 数据可疑、社区无背书，且在其本地 WSL 环境部署时 docker daemon 反复崩溃，无法落地。
- 担心"自动交易"会失控——AI 直接下单、乱改参数、爆仓。
- 对交易机制存在具体疑虑：会不会错过信号？多流派共识如何同步时机？长期不交易时怎么判断系统是"明智等待"还是"坏了"？
- 容易陷入"功能越加越多"的设计过拟合——每想到一个点子就加一个模块，却没有一个经过验证。

一句话：用户需要一套**新手可控、防过拟合、先简单后复杂、人始终把关**的现货量化交易系统方案与落地路径。

## Solution

一套**三层分离**（研究层 / 执行层 / 数据层）的现货量化系统，核心解法：

- **执行层用 Freqtrade**（pip 安装，绕开用户环境的 docker 崩溃问题；52k star 真实、社区活跃、dry-run+实盘+Telegram 原生、纪律机制内置）。
- **研究层用 LLM 生成策略 + 帮助判断**：用户用大白话描述想法，LLM 翻译成 Freqtrade 策略代码；用户用"5 个体检指标 + 3 个防坑问题"判断策略好坏，不需要懂代码。
- **主线先跑通单个简单策略**的完整闭环（生成→回测→dry-run→小资金实盘），**多流派共识是进阶可选**，仅在单策略验证有效且不够稳健时才叠加——从机制上防止设计层过拟合。
- **只做现货**：不做空、不加杠杆、不碰合约与资金费率，最坏结果是亏本金、不会爆仓。
- **可观测性 + LLM 日报**解决"长期不交易看不到效果"的焦虑：记录每周期决策过程，LLM 每天生成健康度日报（只汇报和建议，绝不下单或改参数）。
- **安全边界铁律**：AI 负责"想和判断"，Freqtrade 负责"守纪律执行"，用户负责"决策把关"；研究代码永不与实盘资金共用路径；实盘需双开关显式解锁，默认 paper。

## User Stories

1. 作为不懂代码的个人投资者，我想用大白话描述交易想法，以便 LLM 帮我生成可执行的 Freqtrade 现货策略。
2. 作为新手，我想有一套"5 个体检指标"（最大回撤/胜率/夏普/交易次数/盈亏比），以便不懂策略也能判断一个策略靠不靠谱。
3. 作为新手，我想知道"胜率 >70% 反而要警惕"这类反直觉规则，以便识别过拟合的假象。
4. 作为投资者，我想用"3 个防坑问题"（换币种还赚吗/牛熊分开测/扣费后还赚吗）筛策略，以便快速淘汰垃圾策略。
5. 作为投资者，我想先只跑通一个最简单的策略（如双均线），以便理解完整闭环，而不是一上来就建复杂系统。
6. 作为投资者，我想在多币种、多时段回测同一个策略，以便验证它是"真稳健"还是"只在某段行情靠运气"。
7. 作为投资者，我想让选中的策略在模拟盘（dry-run）长跑 1-3 个月，以便确认模拟表现接近回测（否则说明过拟合）。
8. 作为投资者，我想先用 Testnet 或极小资金（$100-500）做现货实盘，以便在可承受损失下验证真实表现。
9. 作为投资者，我想让系统只做现货三态（买入/持有/卖出），以便永远不会因杠杆爆仓。
10. 作为投资者，我想让 Freqtrade 用代码强制止损（亏 5% 自动砍），以便克服"亏了死扛"的人性弱点。
11. 作为投资者，我想启用账户级熔断（连亏停手/回撤熔断/冷却期），以便避免连亏后加倍下注、恐慌乱操作、手痒频繁交易。
12. 作为投资者，我想限制最多同时持仓数与每笔固定金额，以便防止梭哈和越亏越加码。
13. 作为投资者，我想让 API Key 只开现货交易权限、禁提现、绑 IP 白名单，以便即使密钥泄露也无法被转走资金。
14. 作为投资者，我想让实盘需要"配置项 + 环境变量"双开关才启用、默认 paper，以便杜绝误触发真实下单。
15. 作为投资者，我想让交易所密钥只留在本地部署，以便不把密钥交给任何第三方。
16. 作为进阶用户，我想在单策略不够稳健时叠加 3 个正交流派（趋势/均值回归/量价），以便用多视角互相制衡减少单一模型偏见。
17. 作为进阶用户，我想让流派之间是"正交"的（回答不同问题），以便避免趋势+突破这种重复导致投票失衡。
18. 作为进阶用户，我想让所有流派子信号在同一个锚定决策周期一次性算出，以便避免多实例时机错位且保证可回测。
19. 作为进阶用户，我想用带"死区"的共识决策表，以便只有强共识才交易、避免临界点频繁抖动。
20. 作为投资者，我想让"不交易"成为合法且最常见的决策结果，以便系统在没有机会时明智地空仓而不是硬找理由交易。
21. 作为投资者，我想让"持仓不变=不交易"，以便已持仓且信号未变时继续持有，不重复下单白磨手续费。
22. 作为投资者，我想记录每个决策周期的过程日志（各流派票/信号分/落区/决策/距触发差值），以便在没有交易时也能观测系统。
23. 作为投资者，我想有 5 个"非盈亏"健康指标，以便在长期不交易时判断系统是"健康等待"还是"病态沉默"。
24. 作为投资者，我想能诊断长期不交易的三种原因（阈值太严/流派 bug/行情真没机会），以便对症处理。
25. 作为投资者，我想用 2-3 年回测放大决策样本，以便弥补低频实盘样本太少、得不出可信结论的问题。
26. 作为投资者，我想让 LLM 每天生成一份健康度日报推送到 Telegram，以便不用盯盘也知道系统状态与"为什么没交易"。
27. 作为投资者，我想让 LLM 日报只做"解读和建议"、绝不自动改参数或下单，以便守住人对决策的最终把关权。
28. 作为投资者，我想通过 Telegram bot 收到交易通知并能指挥，以便随时掌控与干预。
29. 作为投资者，我想让 LLM 判读/日报是每天 1 次的批量缓存任务，以便控制 API 成本不失控。
30. 作为投资者，我想让恐惧贪婪指数每天缓存 1 次、只有行情按周期实时拉，以便区分"贵操作缓存、便宜操作实时"降低成本。
31. 作为投资者，我想在回测中扣除手续费（0.1%）和滑点（0.05%），以便回测结果贴近真实、不被虚高欺骗。
32. 作为投资者，我想被明确警告"慎用 hyperopt 暴力调参"，以便不掉进"完美拟合历史、实盘失效"的过拟合陷阱。
33. 作为投资者，我想每加一个流派/参数/旋钮都先问"它验证过吗、比不加强吗"，以便防止功能堆积式的设计过拟合。
34. 作为投资者，我想让共识版只有在回测稳健性优于单策略时才保留，否则回退，以便复杂度必须换来真实收益。
35. 作为投资者，我想理解"信号在 K 线收盘后才判断"会带来延迟但保证可回测/防未来函数，以便接受这个取舍。
36. 作为投资者，我想知道止损/止盈是持仓后实时生效的（不受 K 线延迟影响），以便确信"该止损时不会来不及跑"。
37. 作为投资者，我想优先选做 4h/日线的波段策略，以便信号延迟对我影响很小，同时远离散户必输的高频赛道。
38. 作为投资者，我想有一个清晰的分阶段路线图（阶段 0-5），每阶段有明确验证门槛，以便一步步推进不冒进。
39. 作为投资者，我想所有"收益/年化"类数字都标注为待验证、不做盈利承诺，以便建立正确的风险预期。
40. 作为投资者，我想只用主流币（BTC/ETH）起步，以便规避小币种流动性差、滑点吃光利润的问题。

## Implementation Decisions

**架构**

- 采用三层分离：研究层（LLM 生成+判断+可选共识）/ 执行层（Freqtrade）/ 数据层（行情+情绪+LLM 判读缓存）。三层不共用代码路径；研究产物只是"策略配置/信号/建议"，实盘下单只发生在执行层且需显式晋级。

**执行层（Freqtrade）**

- 用 pip 安装 Freqtrade（不用 docker，规避用户环境 GHCR 大镜像导致的 snap dockerd 崩溃）。
- 纪律通过配置强制：策略级止损（stoploss = -5%）；账户级 Protections（StoplossGuard / MaxDrawdown / CooldownPeriod，需手动开启）；仓位约束（max\_open\_trades=3，stake\_amount 固定金额/笔）。
- 仓位模型（新手版）：每笔固定金额、最多 3 个并发仓位、不做按信号强度动态调仓（进阶再说）。
- dry-run 与 live 为同一套代码，仅由 `dry_run` 开关切换；实盘需配置项 + 环境变量双开关。
- Telegram 原生集成用于通知与指挥。

**研究层**

- 策略生成：LLM 直连（Claude/GPT/DeepSeek 之一）+ 标准提示词模板，输出 Freqtrade IStrategy 代码 + 适用行情说明。
- 策略判断：用户侧使用固定的"5 个体检指标红绿灯"和"3 个防坑问题"，不看单一收益、看跨市场稳健性；胜率过高（>70%）视为过拟合预警。
- 多流派共识（进阶可选）：单个 Freqtrade 策略内实现，锚定单一决策周期（如 4h），每根锚定 K 线收盘时一次性计算 3 个正交流派（趋势/均值回归/量价）子信号，加权得共识分（-1\~+1）。
- 共识决策为现货三态，带死区门槛：
  | 共识分          | 现货决策              |
  | ------------ | ----------------- |
  | ≥ +0.6       | 买入（若空仓）/ 持有（若已持仓） |
  | -0.6 \~ +0.6 | 不交易（死区，维持现状）      |
  | ≤ -0.6       | 卖出持有的币 / 继续空仓     |
  （阈值 0.6 为待验证参数；禁止用 hyperopt 暴力优化。）
- "不交易"是四种合法结果之二（分歧/无信号），且为常态；持仓不变=不交易。

**数据层**

- 行情：交易所现货 API（Binance/OKX），每周期实时拉。
- 情绪：仅保留 alternative.me 恐惧贪婪指数（每天缓存 1 次，可选用于仓位微调）。删除资金费率/套利（合约概念）。
- 成本控制：LLM 判读/日报每天 1 次批量缓存；不实时调 LLM 解读每根 K 线。

**可观测性与 LLM 日报**

- 每个决策周期落一行过程日志（时间/各流派票/信号分/落区/决策/距触发差）。
- 5 个非盈亏健康指标 + 长期不交易三类诊断 + 用回测放大样本。Vibe-Trading
- LLM 日报：每天 1 次读日志→输出结构化日报→推送 Telegram；边界铁律：只解读+建议，绝不自动改参数/下单，建议须用户确认后手动执行。

**安全模型（五道闸）**

- 默认 paper-only；实盘双开关；交易所 Key 只留本地；API Key 最小权限（只现货交易、禁提现、绑 IP）；全程审计日志、异常落盘不静默失败。

**证据分级**

- 全文遵循 事实 / 强判断 / 待验证 分级；所有收益类数字标注待验证，不做盈利承诺。

## Testing Decisions

**测试接缝（seam）——最高、最少**

- 本方案不是传统代码工程，"测试"= 策略验证。**唯一主接缝：Freqtrade 的 backtesting / dry-run 输出**（净值曲线、回撤、成交日志、绩效指标）。所有策略——无论单策略还是共识版——都在这一个接缝上被验证，保证"回测与实盘同一引擎"。
- 好的验证只看**外部行为**（策略在给定历史数据上的成交与绩效），不看内部实现细节（具体指标怎么算）。

**验证内容**

- 单策略：在多币种（BTC/ETH）× 多时段（牛/熊/震荡）回测，对照 5 指标 + 3 防坑问题；再 dry-run 长跑对比回测一致性。
- 共识版（若启用）：同一接缝回测，且必须**稳健性优于单策略**方可保留，否则回退。
- 可观测性：通过决策过程日志验证"每周期都有决策记录"（含不交易），用 5 健康指标区分健康等待 vs 病态沉默。

**防过拟合验证纪律**

- 样本外时段验证；扣除手续费+滑点；慎用 hyperopt 且必须留样本外；低频用回测放大样本弥补实盘样本不足。

**Prior art**

- 复用 Freqtrade 官方 backtesting/dry-run 与 Protections 机制作为验证基础设施，不自建回测引擎。

## Out of Scope

- 合约、杠杆、做空、永续合约资金费率、套利策略（本版锁定现货）。
- 高频/剥头皮交易（散户在延迟与基础设施上必输）。
- 链上数据、巨鲸监控、推特/新闻 NLP 情感分析（信息过载/噪音大，进阶或永不）。
- 按信号强度动态调仓、Walk-forward、蒙特卡洛验证（进阶，当前是负担）。
- QuantDinger 本地部署（star 存疑 + 用户环境 docker 崩溃，已放弃；仅借鉴其架构思想）。
- Vibe-Trading（其"聊天生成策略 + 回测"两项职责已分别由 LLM 直连和 Freqtrade 内置回测覆盖；为减少组件依赖、避免双回测引擎结果不一致、并让 LLM 直接产出 Freqtrade 格式，本版不采用。其 452 因子库/可视化等能力对新手单策略主线非必需，进阶再议）。
- A 股 / 港美股 / 外汇（本版聚焦加密现货；vnpy 等留作未来另议）。
- 任何盈利承诺或具体收益预测。

## Further Notes

- **反过拟合是本 spec 的第一性原则**：不仅针对交易策略，也针对方案设计本身——防止"每轮对话加一个功能"的功能堆积。共识、情绪、日报等都被刻意降级为可选或精简，主线只要求跑通一个简单策略。
- **人始终在回路中**：AI 只做生成、判断辅助、日报建议；下单与改参数由 Freqtrade 机制和用户把关。这是安全边界，不可让渡。

- 详细技术设计与代码级配置见 [AI量化交易系统技术方案.md](file:///f:/repos/ai_doc/AI量化交易系统技术方案.md) v2.0。
- 开始实施（阶段 0）前需用户拍板：LLM 选型、Testnet 还是主网、是否现在配 Telegram bot、起步品种。

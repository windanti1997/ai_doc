AI 编码不是缺模型，是缺脚手架

你有没有过这种经历？

早上打开 Claude Code，说「帮我把用户邀请功能做了」。AI 噼里啪啦写了一堆文件，你一看，嗯，大体方向对。

下午换 Cursor 调 UI 细节，发现上午写的 API 字段名和前端对不上。你叹了口气，又得重新解释一遍。

第二天开新会话，AI 问你：「这个项目用什么 ORM？测试怎么跑？」你盯着屏幕，突然不知道该生气还是该无奈——这些你昨天不是刚说过吗？

这不是模型不够聪明。

这是你和 AI 之间，缺了一层脚手架。

01

PART


AI 编码 10% 是写代码

PAIN POINT · 90% 是重新解释技术栈

我身边很多人用 AI 编程，都陷入了一个怪圈：

模型越来越强，Claude 从 3.5 换到 Opus，Cursor 从 v0.10 用到 v1.0，但效率提升的体感越来越弱。

为什么？

因为瓶颈不在「AI 会不会写代码」，而在「每次开新会话，你得重新讲一遍你的项目」。

你的项目有自己的脾气


后端层

用什么框架，数据库怎么分层，错误处理怎么返回，日志打哪些字段


前端层

组件怎么命名，样式用 CSS-in-JS 还是 Tailwind，状态管理用什么方案


工程层

测试怎么组织，提交怎么写 message，哪些目录不能动，哪些文件是生成的

这些东西，你跟一个新同事讲一遍要一天。跟 AI 讲一遍？每次会话都要讲一遍。

于是你写了 CLAUDE.md，写了 .cursorrules，写了 AGENTS.md。

然后呢？

文件越来越长，从 100 行写到 500 行，再到 1000 行。AI 真的读完了吗？你不确定。反正它有时候还是会忘。

更糟的是团队场景

张三用 Claude Code，李四用 Cursor，王五用 Codex——每个人的规则文件各写各的；一个人踩过的坑，另一个人的 AI 照样踩；Reviewer 每次都要重复同样的评论：「这里要加 loading 状态」「错误格式不对」「我们已经有这个工具函数了」。

这就是当下 AI 编程最大的浪费

规则散落在各处，经验无法沉淀，每次会话从零开始。

PART


我试过的三种方案

COMPARISON · 各有各的问题

这一年我前前后后试了三类工具，每一类都解决了一部分问题，但又留下了新的坑。

第一种：流程强化型 —— Superpowers

这是最流行的一类，代表就是 Superpowers（170K+ stars）。核心思路是：给 AI 套上一套严格的工程流程——先头脑风暴，再写设计文档，再拆计划，再 TDD，最后 Code Review。

✦ 优点很明显

有了流程约束，AI 不会一上来就瞎写代码；头脑风暴环节能帮你把模糊的需求想清楚；Sub-agent 隔离，减少上下文污染。

！但用了一个月我就停了 🕳

第一，太重了。 哪怕只是改一个按钮文案，它也要跟你确认三轮、走五步流程。小任务的 overhead 比收益还大。

第二，上下文吃紧。 Superpowers 自己的框架提示就占了一大块，长对话跑着跑着上下文就满了，自动压缩后细节丢得厉害。

第三，没有记忆。 这次确认的规则、踩过的坑，下次会话照样要重来。它解决的是「单次任务怎么做」，没解决「跨会话怎么积累」。

第二种：规格沉淀型 —— Spec Kit

第二类是 Spec Kit / OpenSpec 这种，核心是「先对齐，再实现」。把项目规范、设计决策、编码标准都写成结构化文档，AI 编码前先读。

✦ 优点

规范显式化了，不再只存在于老员工脑子里；文档可以被 Review，可以版本化；团队有了统一的参照物。

！问题也很现实 🕳

第一，写了不代表 AI 会读。 你把规范写得再好，AI 上下文一长就容易忽略。你没法强制它每次都先读完再动手。

第二，只有规范，没有流程闭环。 它告诉你「应该怎么样」，但不负责「怎么确保做到」。AI 写代码时有没有遵守规范，还是要靠人去查。

第三，缺少需求澄清环节。 Spec 是结果，但从「模糊想法」到「清晰 spec」的过程，它帮不上忙。

第三种：多 Agent 编排型

第三类走得更远，直接做多 Agent 分工：planner 负责规划，executor 负责写代码，reviewer 负责检查，debugger 负责修 bug。代表是 oh-my-claudecode。

！听起来很美，但实际用下来 🕳

第一，太复杂了。 29 个 agents、35 个 skills、20 个 hooks、11 个生命周期事件。出了问题你根本不知道在哪一层。

第二，绑定太深。 严重依赖 Claude Code 的 Plugin 体系，换个工具就全废了。

第三，简单任务过度编排。 改个配置文件也要四个 Agent 来回传话，效率反而更低。

三类方案对比表

维度 Superpowers Spec Kit 多Agent编排
核心问题 单次任务流程 规范文档化 复杂任务分工
上手门槛 低 中 高
系统复杂度 中 低 高
持久化记忆 ✗ 部分 ✗
多平台支持 4个左右 不限 基本绑定

三种方案，三个方向，各有各的道理。但我总觉得还差一点什么。

直到我遇到了 Trellis。

Trellis 是什么

CORE · AI 编码助手的辅助轮

第一次看到 Trellis 的定位，我愣了一下。

它说自己是「AI 编码助手的辅助轮」。

不是「更强的工作流」，不是「更好的 spec 工具」，而是辅助轮。

这个比喻很妙

辅助轮不帮你骑车，但它能保证你不往沟里歪

AI 写代码也是一样——模型本身已经很能写了，问题是它容易跑偏、容易忘事、容易不按规矩来。

Trellis 做的事情，就是给 AI 套上三层辅助轮。

Spec 规范层

你的规矩，AI 每次都遵守

Task 任务层

每个任务都是可审查的工作单元

Workspace 记忆层

解决 AI 的「会话失忆症」

三层架构，层层递进

第一层：Spec 规范层

Trellis 把项目规范存在 .trellis/spec/ 目录下，按模块分文件。

. . . 目录结构

.trellis/spec/

├── backend/

│ ├── database-guidelines.md

│ ├── error-handling.md

│ └── api-validation.md

├── frontend/

│ ├── component-guidelines.md

│ └── state-management.md

└── guides/

└── testing-practices.md

这跟你手写 CLAUDE.md 有什么区别？

三个关键区别：



区别 1

按需加载，不是一股脑全塞

Trellis 会根据当前任务类型，只把相关的 spec 注入上下文。做后端就注入后端规范，改前端就注入前端规范。不会让 1000 行规则占满上下文。



区别 2

自动注入，不用手动提醒

支持 Hook 的平台，Trellis 会在会话启动时自动把规范注入。不需要你每次说「先读一下 CLAUDE.md」。


区别 3

自更新，越用越懂你

每次任务结束后，Trellis 会检查有没有产生新的最佳实践，如果有就自动更新 spec 库。用得越多，AI 越贴合你的项目。

第二层：Task 任务层

Trellis 的 Task 不只是一行任务描述。

每个 Task 是一个物理目录，包含完整的上下文。

这件事的意义，比你想象的大。

首先，需求从聊天记录里搬到了文件里。不再是「我跟 AI 聊过，它应该知道」。而是 prd.md 清清楚楚写着：目标是什么、scope 内有什么、scope 外有什么、验收标准是什么。谁都能看，谁都能 Review。

其次，上下文有了明确边界。实现阶段读什么、检查阶段对照什么，都列出来了。AI 不会东看西看把自己看糊涂，也不会漏掉关键约束。

第三，任务有生命周期。Planning → In Progress → Check → Finish → Archived。每个阶段该做什么、不该做什么，清清楚楚。不会出现「写着写着代码突然又去改需求」的情况。

第三层：Workspace 记忆层

这是 Trellis 最打动我的一层。

每次会话结束后，Trellis 会把这次做了什么、遇到了什么问题、下一步计划是什么，写进 workspace journal。

第二天开新会话，AI 先读 journal。

就这么一个简单的动作，解决了最大的痛点

我自己用下来最直观的感受是：

以前开新会话，前 20 分钟基本都是在「让 AI 回忆起来我们在干什么」。现在不用了，它读一下 journal，立刻就接上了上次的思路。

更重要的是团队场景。

一个人踩过的坑、总结的规范，写进 spec，提交到 Git，整个团队的 AI 都能受益。不再是「张三知道但李四不知道」，而是「写一次，所有人的 AI 都学会了」。

这是 Trellis 最核心的团队价值

把个人经验，变成团队资产。

真实案例

CASES · 3 个 Good + 2 个 Bad

光说概念太虚，讲讲我实际用下来的真实感受。



GOOD 01

老项目接入，规范从 0 到有

LEGACY PROJECT · 从真实代码提取模式

我们有一个跑了三年的 SaaS 项目，代码量不小，各种隐含模式散落在历史 PR 和老员工记忆里。以前让 AI 改东西，基本上是「改一次踩一次坑」。

接入 Trellis 的第一步，我们没有自己写 spec。直接让 AI 扫描代码库，从真实代码里提取模式，自动生成第一版 spec。然后团队花了半天 Review，删掉了那些「想当然」的规则，只保留能找到真实源码例子的。

效果非常明显。之前做一个需求，我可能要在 Review 时指出 5-6 个「不符合项目规范」的问题。接入后，通常只有 1-2 个，而且大多是 spec 还没覆盖到的新场景。



GOOD 02

团队推广，规范不再靠口口相传

TEAM COLLABORATION · 个人经验变团队资产

推广到团队是最意外的收获。我们团队 8 个人，用 Claude Code 的、用 Cursor 的、用 Codex 的都有。以前每个人的 AI 助手各干各的，风格不统一，规范不一致。

接入 Trellis 后，`.trellis/spec/` 目录提交到 Git，所有人拉代码就能拿到最新规范。

Review 也变了。以前 Review 代码，你得跟新人解释「为什么我们这里要这么写」。现在不用了，直接指一下 spec 里的某一条：「看，这里有规定。」



GOOD 03

修复反复出现的 Bug，从治标到治本

BUG PREVENTION · 同样的坑不踩第二次

有一个 SessionStart hook 崩溃的 bug 折腾了我们很久。现象是：用户说 Python 是 3.11，但 hook 子进程里跑的是系统自带的 3.9，碰到语法就挂。

如果是以前，修完就完了。最多在团队群里说一句「大家注意一下 Python 版本问题」。

用 Trellis 之后，我们走了完整的 break-loop 流程：先修 bug、做根因分析、沉淀成 spec、加进 check 清单。同样的坑，以后不会再踩第二次。



BAD 01

一开始就想写完整的 spec

COMMON MISTAKE · 我犯的第一个错误

刚接入 Trellis 时，我雄心勃勃地想把所有规范都写好，再开始用。花了整整两天，填了 20 多个模板文件。

结果呢？一半是我脑补的「理想规范」，跟实际代码对不上。AI 按 spec 写，写出来的东西跟现有项目风格不搭。最后还得推翻重来。

💡 正确姿势：先从真实代码里提取，不要预先设计整个体系。用到哪写到哪，写一条是一条。空 spec 比没有 spec 危害还大。


BAD 02

所有任务都套 Trellis 流程

OVER ENGINEERING · 小任务不值得

第二个坑：什么任务都要建 Trellis task，走完整流程。改个文案、调个样式、加个配置项，本来 5 分钟的事，走 Trellis 流程要 15 分钟。得不偿失。

Trellis 自己也意识到了这点，它有三种模式：简单对话、Inline 小任务、完整 Trellis 任务。

💡 不是所有任务都值得上完整流程。小改动直接 inline 就行，Trellis 是给「值得认真做的任务」准备的。

不同场景怎么用

SCENARIOS · 5 个典型场景

根据官方文档和我自己的实践，整理了几个典型场景的用法：

SCENE 01

从零开始新项目

先跑起来，边做边补 spec。第 1 周先写核心技术栈的 spec（2-3 个文件），第 2 周起每做一个新模块补充对应 spec。

关键：不要等 spec 写全了再开始写代码。

SCENE 02

接入存量老项目

从真实代码提取，不要凭空设计。第一步让 AI 扫描代码库生成第一版 spec，第二步团队 Review 删掉脑补的，第三步找一个小需求试点。

关键：每条 spec 必须能找到源码依据。

SCENE 03

交付产品功能

跨层次的功能（产品+API+数据+UI），用 Trellis 让 scope、spec、实现、检查保持一致。从自然语言描述到 prd.md，再到 implement → check → update-spec，走完完整闭环。

SCENE 04

重构老模块

先列「行为必须保持不变」的清单，写进 PRD。加 characterization tests 跑通基线。一次只抽取一个职责，每轮都要 Review diff。重构完更新 spec。

SCENE 05

减少重复 Review 反馈

Reviewer 总是写同类评论？收集最近的 PR 评论，归类重复模式，每条规则配 good/bad 示例写进 spec。下次任务自动检查这些规则。

个人 vs 团队，侧重点不同

维度 个人使用 团队使用
最大价值 会话记忆，不用重复解释 规范共享，一人总结全队受益
核心投入 workspace journals spec 库，持续维护
起步成本 低，跑两个任务就上手 中，需要团队达成共识
见效时间 当天就能感受到 1-2 周后

常见问题

FAQ · 几个你可能关心的问题

Q1：Trellis 支持哪些平台？

官方支持 16 个：Claude Code、Cursor、OpenCode、Codex、Kiro、Kilo、Gemini CLI、Devin、Qoder、CodeBuddy、GitHub Copilot、Droid、Pi Agent、Reasonix、ZCode 等。

但说实话，体验最好的还是 Claude Code，因为它的 Hook 系统最完善，自动化程度最高。Cursor 和 OpenCode 次之。

Q2：AGPL 协议会不会有问题？

这是个好问题。Trellis 用的是 AGPL-3.0。

如果你只是用它来开发自己的项目，不管是个人还是公司，完全没问题。AGPL 管的是「你修改了 Trellis 本身，并且把修改后的版本作为服务提供给别人」的情况。

但如果你公司的法务对 AGPL 很敏感，或者你们想基于 Trellis 做商业产品，那确实需要评估一下。

Q3：会不会让 AI 编程变慢？

短期看，是的。第一次用，你得花时间初始化、写 spec、熟悉流程。前几个任务可能比直接写还慢。

但长期看，是快的。因为规范在积累、记忆在沉淀、重复沟通在减少。做得越多，AI 越懂你的项目，每一次任务的起步成本越低。

体感曲线：第 1 周变慢 → 第 2 周持平 → 第 3 周开始变快 → 第 4 周明显提速

Q4：跟 CLAUDE.md 比，到底好在哪？

如果你项目很小、只有一个人用、规则文件也就一两百行，那 CLAUDE.md 够用了。

Trellis 的价值体现在：规则多了单文件会变单体、多平台使用要维护多份、团队协作经验无法沉淀、跨会话记忆每次从零开始。

这些问题，单文件方案解决不了。

写在最后

CONCLUSION · 模型是发动机

最后说句实在话。

Trellis 不是银弹。

它不能把一个模糊的需求自动变成完美代码。它不能替代产品经理，不能替代架构师，也不能替代 Code Review。

它做的事情很朴素：


把你的规矩记下来


把做过的事记下来


把一个人的经验，变成全队的资产

这件事不性感，但很重要。

AI 编程的下半场，比的是谁的 AI 更懂你的项目

AI 编程的早期，大家都在比「模型谁更强」「谁生成代码更快」。但现在，模型能力的差距在缩小。真正拉开差距的，是你能不能让 AI 稳定地、持续地、按你的标准输出高质量代码。

模型是发动机。但只有发动机，没有方向盘、没有刹车、没有导航，车开不快，也开不远。

Trellis 就是那套方向盘+刹车+导航。

一句话总结

它不会让车的马力变大，但它能让你敢把油门踩下去

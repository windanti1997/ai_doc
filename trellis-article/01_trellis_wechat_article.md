# 用了三个月 Trellis，我终于敢让 AI 独立干活了

副标题：从 Superpowers 到 Spec Kit 再到 Trellis，我踩过的 AI 编程工作流的坑。

建议封面文案：AI 编码不是缺模型，是缺脚手架。

---

你有没有过这种经历？

早上打开 Claude Code，说"帮我把用户邀请功能做了"。AI 噼里啪啦写了一堆文件，你一看，嗯，大体方向对。

下午换 Cursor 调 UI 细节，发现上午写的 API 字段名和前端对不上。你叹了口气，又得重新解释一遍。

第二天开新会话，AI 问你："这个项目用什么 ORM？测试怎么跑？"你盯着屏幕，突然不知道该生气还是该无奈——这些你昨天不是刚说过吗？

这不是模型不够聪明。

这是你和 AI 之间，缺了一层脚手架。

## AI 编码 10% 是写代码，90% 是重新解释技术栈

我身边很多人用 AI 编程，都陷入了一个怪圈：

模型越来越强，Claude 从 3.5 换到 Opus，Cursor 从 v0.10 用到 v1.0，但效率提升的体感越来越弱。

为什么？

因为瓶颈不在"AI 会不会写代码"，而在"每次开新会话，你得重新讲一遍你的项目"。

你的项目有自己的脾气：
- 后端用什么框架，数据库怎么分层
- 前端组件怎么命名，样式用 CSS-in-JS 还是 Tailwind
- 错误处理怎么返回，日志打哪些字段
- 测试怎么组织，提交怎么写 message
- 哪些目录不能动，哪些文件是生成的不能手改

这些东西，你跟一个新同事讲一遍要一天。跟 AI 讲一遍？每次会话都要讲一遍。

于是你写了 `CLAUDE.md`，写了 `.cursorrules`，写了 `AGENTS.md`。

然后呢？

文件越来越长，从 100 行写到 500 行，再到 1000 行。AI 真的读完了吗？你不确定。反正它有时候还是会忘。

更糟的是团队场景。
- 张三用 Claude Code，李四用 Cursor，王五用 Codex
- 每个人的规则文件各写各的
- 一个人踩过的坑，另一个人的 AI 照样踩
- Reviewer 每次都要重复同样的评论："这里要加 loading 状态""错误格式不对""我们已经有这个工具函数了"

这就是当下 AI 编程最大的浪费：**规则散落在各处，经验无法沉淀，每次会话从零开始。**

## 我试过的三种方案，各有各的问题

这一年我前前后后试了三类工具，每一类都解决了一部分问题，但又留下了新的坑。

### 第一种：流程强化型 —— Superpowers

这是最流行的一类，代表就是 Superpowers（170K+ stars）。核心思路是：给 AI 套上一套严格的工程流程 —— 先头脑风暴，再写设计文档，再拆计划，再 TDD，最后 Code Review。

**优点很明显：**
- 有了流程约束，AI 不会一上来就瞎写代码
- 头脑风暴环节能帮你把模糊的需求想清楚
- Sub-agent 隔离，减少上下文污染

**但用了一个月我就停了，原因有三个：**

第一，**太重了**。哪怕只是改一个按钮文案，它也要跟你确认三轮、走五步流程。小任务的 overhead 比收益还大。

第二，**上下文吃紧**。Superpowers 自己的框架提示就占了一大块，长对话跑着跑着上下文就满了，自动压缩后细节丢得厉害。

第三，**没有记忆**。这次确认的规则、踩过的坑，下次会话照样要重来。它解决的是"单次任务怎么做"，没解决"跨会话怎么积累"。

### 第二种：规格沉淀型 —— Spec Kit

第二类是 Spec Kit / OpenSpec 这种，核心是"先对齐，再实现"。把项目规范、设计决策、编码标准都写成结构化文档，AI 编码前先读。

**优点：**
- 规范显式化了，不再只存在于老员工脑子里
- 文档可以被 Review，可以版本化
- 团队有了统一的参照物

**问题也很现实：**

第一，**写了不代表 AI 会读**。你把规范写得再好，AI 上下文一长就容易忽略。你没法强制它每次都先读完再动手。

第二，**只有规范，没有流程闭环**。它告诉你"应该怎么样"，但不负责"怎么确保做到"。AI 写代码时有没有遵守规范，还是要靠人去查。

第三，**缺少需求澄清环节**。Spec 是结果，但从"模糊想法"到"清晰 spec"的过程，它帮不上忙。

### 第三种：多 Agent 编排型 —— oh-my-claudecode

第三类走得更远，直接做多 Agent 分工：planner 负责规划，executor 负责写代码，reviewer 负责检查，debugger 负责修 bug。

**听起来很美，但实际用下来：**

第一，**太复杂了**。29 个 agents、35 个 skills、20 个 hooks、11 个生命周期事件。出了问题你根本不知道在哪一层。

第二，**绑定太深**。严重依赖 Claude Code 的 Plugin 体系，换个工具就全废了。

第三，**简单任务过度编排**。改个配置文件也要四个 Agent 来回传话，效率反而更低。

我把这三类方案的对比如下：

| 维度 | Superpowers | Spec Kit | 多 Agent 编排 |
|---|---|---|---|
| 解决的核心问题 | 单次任务的流程纪律 | 规范的文档化沉淀 | 复杂任务的分工协作 |
| 上手门槛 | 低 | 中 | 高 |
| 系统复杂度 | 中 | 低 | 高 |
| 持久化记忆 | ✗ | 部分 | ✗ |
| 多平台支持 | 4 个左右 | 不限（纯文档） | 基本绑定 Claude |
| 适合场景 | 原型/小团队 | 规范化团队 | 超复杂多 Agent 任务 |

三种方案，三个方向，各有各的道理。但我总觉得还差一点什么。

直到我遇到了 Trellis。

## Trellis 不是又一个工作流框架，它是 AI 的项目记忆系统

第一次看到 Trellis 的定位，我愣了一下。

它说自己是"AI 编码助手的辅助轮"。

不是"更强的工作流"，不是"更好的 spec 工具"，而是**辅助轮**。

这个比喻很妙。辅助轮不帮你骑车，但它能保证你不往沟里歪。AI 写代码也是一样 —— 模型本身已经很能写了，问题是它容易跑偏、容易忘事、容易不按规矩来。

Trellis 做的事情，就是给 AI 套上三层辅助轮：

### 第一层：Spec 规范层 —— 你的规矩，AI 每次都遵守

Trellis 把项目规范存在 `.trellis/spec/` 目录下，按模块分文件：

```
.trellis/spec/
├── backend/
│   ├── database-guidelines.md
│   ├── error-handling.md
│   ├── logging-guidelines.md
│   └── api-validation.md
├── frontend/
│   ├── component-guidelines.md
│   ├── state-management.md
│   └── styling-conventions.md
└── guides/
    ├── code-review-checklist.md
    └── testing-practices.md
```

这跟你手写 `CLAUDE.md` 有什么区别？

三个关键区别：

**第一，按需加载，不是一股脑全塞。**
Trellis 会根据当前任务类型，只把相关的 spec 注入上下文。做后端就注入后端规范，改前端就注入前端规范。不会让 1000 行规则占满上下文。

**第二，自动注入，不用手动提醒。**
支持 Hook 的平台（Claude Code、Cursor、OpenCode 等），Trellis 会在会话启动时自动把规范注入。不需要你每次说"先读一下 CLAUDE.md"。

**第三，自更新，越用越懂你。**
每次任务结束后，Trellis 会检查有没有产生新的最佳实践，如果有就自动更新 spec 库。用得越多，AI 越贴合你的项目。

一条好的 spec 长什么样？Trellis 官方文档里给了一个标准：

> 具体、有示例、说清为什么、能追溯到源文件。

不是"代码应该整洁规范"这种空话，而是：

```
## API Input Validation

所有 API route 必须在调用 service 之前用 Zod 校验请求体。
Schema 与 route 文件同目录。

// Bad
const user = await userService.create(req.body);

// Good
const input = CreateUserSchema.parse(req.body);
const user = await userService.create(input);
```

有正反例，有具体路径，有明确规则。AI 看到就知道该怎么做，Reviewer 看到也知道该怎么查。

### 第二层：Task 任务层 —— 每次任务都是一个可审查的工作单元

Trellis 的 Task 不只是一行任务描述。

每个 Task 是一个物理目录，包含完整的上下文：

```
.trellis/tasks/07-08-team-invitations/
├── task.json          # 任务元数据、状态
├── prd.md             # 产品需求文档
├── design.md          # 技术设计（复杂任务才有）
├── implement.md       # 执行计划（复杂任务才有）
├── implement.jsonl    # 实现阶段需要读哪些文件
├── check.jsonl        # 检查阶段需要对照哪些规范
└── research/          # 调研产出
```

这件事的意义，比你想象的大。

**首先，需求从聊天记录里搬到了文件里。**
不再是"我跟 AI 聊过，它应该知道"。而是 prd.md 清清楚楚写着：目标是什么、scope 内有什么、scope 外有什么、验收标准是什么。谁都能看，谁都能 Review。

**其次，上下文有了明确边界。**
实现阶段读什么、检查阶段对照什么，都用 JSONL 列出来了。AI 不会东看西看把自己看糊涂，也不会漏掉关键约束。

**第三，任务有生命周期。**
Planning → In Progress → Check → Finish → Archived。每个阶段该做什么、不该做什么，清清楚楚。不会出现"写着写着代码突然又去改需求"的情况。

### 第三层：Workspace 记忆层 —— 解决 AI 的"会话失忆症"

这是 Trellis 最打动我的一层。

每次会话结束后，Trellis 会把这次做了什么、遇到了什么问题、下一步计划是什么，写进 `.trellis/workspace/` 下的 journal 文件。

第二天开新会话，AI 先读 journal。

就这么一个简单的动作，解决了最大的痛点：**不用每次重新解释项目背景。**

我自己用下来最直观的感受是：

以前开新会话，前 20 分钟基本都是在"让 AI 回忆起来我们在干什么"。现在不用了，它读一下 journal，立刻就接上了上次的思路。

更重要的是团队场景。

一个人踩过的坑、总结的规范，写进 spec，提交到 Git，整个团队的 AI 都能受益。
不再是"张三知道但李四不知道"，而是"写一次，所有人的 AI 都学会了"。

这是 Trellis 最核心的团队价值：**把个人经验，变成团队资产。**

## 四个月用下来，我总结了 3 个 Good Case 和 2 个 Bad Case

光说概念太虚，讲讲我实际用下来的真实感受。

### Good Case 1：老项目接入，规范从 0 到有

我们有一个跑了三年的 SaaS 项目，代码量不小，各种隐含模式散落在历史 PR 和老员工记忆里。

以前让 AI 改东西，基本上是"改一次踩一次坑"：
- 错误处理格式不对
- 日志没按规范打
- 用了已经废弃的 tlog 而不是新的 slog
- SQL 写法有性能问题

接入 Trellis 的第一步，我们没有自己写 spec。
直接让 AI 扫描代码库，从真实代码里提取模式，自动生成第一版 spec。然后团队花了半天 Review，删掉了那些"想当然"的规则，只保留能找到真实源码例子的。

效果非常明显。
之前做一个需求，我可能要在 Review 时指出 5-6 个"不符合项目规范"的问题。接入后，通常只有 1-2 个，而且大多是 spec 还没覆盖到的新场景。

更妙的是，每次发现新的规范缺口，直接用 `/trellis:update-spec` 加上去。下次 AI 自己就知道了。

### Good Case 2：团队推广，规范不再靠口口相传

推广到团队是最意外的收获。

我们团队 8 个人，用 Claude Code 的、用 Cursor 的、用 Codex 的都有。以前每个人的 AI 助手各干各的，风格不统一，规范不一致。

接入 Trellis 后，`.trellis/spec/` 目录提交到 Git，所有人拉代码就能拿到最新规范。

Review 也变了。
以前 Review 代码，你得跟新人解释"为什么我们这里要这么写"。现在不用了，直接指一下 spec 里的某一条："看，这里有规定。"

新同事入职也更快了。
以前要花一周熟悉项目规范，现在让他先读 `.trellis/spec/` 里的核心文件，再跟着 Trellis 流程做两个小任务，基本上手了。

### Good Case 3：修复反复出现的 Bug，从"治标"到"治本"

有一个 SessionStart hook 崩溃的 bug 折腾了我们很久。
现象是：用户说 Python 是 3.11，但 hook 子进程里跑的是系统自带的 3.9，碰到 `str | None` 这种 PEP 604 语法就挂。

如果是以前，修完就完了。最多在团队群里说一句"大家注意一下 Python 版本问题"。

用 Trellis 之后，我们走了完整的 break-loop 流程：
1. 先修 bug，加回归测试
2. 做根因分析：为什么会发生？因为 hook 子进程的 PATH 极简，不是用户 shell 的 PATH
3. 沉淀成 spec：所有 hook 脚本必须兼容 Python 3.9，或者开头做版本检查
4. 加进 check 清单：以后每次改 hook，自动检查版本兼容性

同样的坑，以后不会再踩第二次。

这才是真正的"长经验"。

### Bad Case 1：一开始就想写完整的 spec

这是我犯的第一个错误。

刚接入 Trellis 时，我雄心勃勃地想把所有规范都写好，再开始用。花了整整两天，填了 20 多个模板文件。

结果呢？
一半是我脑补的"理想规范"，跟实际代码对不上。
AI 按 spec 写，写出来的东西跟现有项目风格不搭。
最后还得推翻重来。

**正确的姿势是：先从真实代码里提取，不要预先设计整个体系。**
先用起来，在做任务的过程中逐步补充 spec。用到哪写到哪，写一条是一条。空 spec 比没有 spec 危害还大，因为它会给你虚假的安全感。

### Bad Case 2：所有任务都套 Trellis 流程

第二个坑：什么任务都要建 Trellis task，走完整流程。

改个文案、调个样式、加个配置项，本来 5 分钟的事，走 Trellis 流程要 15 分钟。得不偿失。

Trellis 自己也意识到了这点，它有三种模式：
- **简单对话**：不写仓库，纯聊天，不建任务
- **Inline 小任务**：小范围改动，可以跳过 Trellis
- **完整 Trellis 任务**：多文件、跨模块、需要持久规划的才建任务

**不是所有任务都值得上完整流程。**
小改动直接 inline 就行，Trellis 是给"值得认真做的任务"准备的。

## 不同场景怎么用 Trellis，一张图说清楚

根据官方文档和我自己的实践，整理了几个典型场景的用法：

### 场景 1：从零开始新项目

**适合时机**：新产品、新服务、新工具
**Trellis 的价值**：在代码扩散前把早期决策定下来

怎么做：
1. `trellis init` 初始化，生成默认 spec 模板
2. 先做 bootstrap 任务，讨论技术栈和项目结构
3. 优先填充用得到的 spec，不要填满所有模板
4. 从一个最小的端到端任务开始跑

### 场景 2：接入存量项目

**适合时机**：老仓库、接手项目、历史包袱重
**Trellis 的价值**：从真实代码提取模式，不打断业务开发

怎么做：
1. `trellis init` 初始化
2. 让 AI 扫描代码库，从现有代码提取第一版 spec
3. 每条规范必须能引用真实源文件，不能写"想当然"的规则
4. 找一个小需求试点，观察效果

### 场景 3：交付产品功能

**适合时机**：跨层次的功能（产品+API+数据+UI）
**Trellis 的价值**：让 scope、spec、实现、检查保持一致

怎么做：
1. 用自然语言描述需求，trellis-brainstorm 引导澄清
2. 写 prd.md，明确 in-scope / out-of-scope / 验收标准
3. 复杂任务拆 subtask，但要在同一个 task 里保持上下文
4. implement → check → update-spec，走完完整闭环

### 场景 4：重构老模块

**适合时机**：代码能跑，但难以安全修改
**Trellis 的价值**：在保持行为不变的前提下，让结构可审查

怎么做：
1. 先列"行为必须保持不变"的清单，写进 PRD
2. 加 characterization tests，跑通基线
3. 一次只抽取一个职责，每轮都要 Review diff
4. 重构完更新 spec，把新的模块边界沉淀下来

### 场景 5：减少重复 Review 反馈

**适合时机**：Reviewer 总是写同类评论
**Trellis 的价值**：把 Review 规则提升为共享仓库上下文

怎么做：
1. 收集最近的 PR 评论，归类重复模式
2. 每条规则配 good/bad 示例，写进 spec
3. 下次任务自动检查这些规则
4. 跑几个任务后回看：哪些规则没抓到、哪些产生噪音，持续优化

### 个人 vs 团队，侧重点不同

| 维度 | 个人使用 | 团队使用 |
|---|---|---|
| 最大价值 | 会话记忆，不用每次重新解释 | 规范共享，一个人总结全队受益 |
| 核心投入 | workspace journals，记录进度和坑 | spec 库，持续维护和 Review |
| 起步成本 | 低，跑两个任务就上手了 | 中，需要团队达成共识 |
| 见效时间 | 当天就能感受到 | 1-2 周后规范积累到一定量 |

## 几个你可能关心的问题

### Q1：Trellis 支持哪些平台？

官方支持 16 个：Claude Code、Cursor、OpenCode、Codex、Kiro、Kilo、Gemini CLI、Antigravity、Devin、Qoder、CodeBuddy、GitHub Copilot、Droid、Pi Agent、Reasonix、ZCode。

另外，任何读取 `.agents/skills/` 规范的 Agent（Amp、Cline、Deep Agents、Firebender、Kimi Code CLI、Warp 等）都能用。

但说实话，**体验最好的还是 Claude Code**，因为它的 Hook 系统最完善，自动化程度最高。Cursor 和 OpenCode 次之。其他平台可能需要手动触发一些命令。

### Q2：AGPL 协议会不会有问题？

这是个好问题。Trellis 用的是 AGPL-3.0。

如果你只是**用它来开发自己的项目**，不管是个人还是公司，完全没问题。AGPL 管的是"你修改了 Trellis 本身，并且把修改后的版本作为服务提供给别人"的情况。

但如果你公司的法务对 AGPL 很敏感，或者你们想基于 Trellis 做商业产品，那确实需要评估一下。

### Q3：会不会让 AI 编程变慢？

短期看，是的。
第一次用，你得花时间初始化、写 spec、熟悉流程。前几个任务可能比直接写还慢。

但长期看，是快的。
因为规范在积累、记忆在沉淀、重复沟通在减少。做得越多，AI 越懂你的项目，每一次任务的起步成本越低。

我自己的体感是：
- 第 1 周：变慢了，因为在搭架子
- 第 2 周：持平，偶尔还要查文档
- 第 3 周：开始变快，常见问题 AI 自己都知道了
- 第 4 周：明显提速，特别是做第二个、第三个同类需求时

### Q4：跟 CLAUDE.md / .cursorrules 比，到底好在哪？

如果你项目很小、只有一个人用、规则文件也就一两百行，那 CLAUDE.md 够用了。

Trellis 的价值体现在：
- 规则多了之后，单文件会变成单体，AI 看不过来
- 多平台使用，每个工具要写一份规则，维护成本高
- 团队协作，个人经验无法沉淀成团队资产
- 跨会话记忆，每次开新会话都要重新来

这些问题，单文件方案解决不了。

## 最后说句实在话

Trellis 不是银弹。

它不能把一个模糊的需求自动变成完美代码。它不能替代产品经理，不能替代架构师，也不能替代 Code Review。

它做的事情很朴素：
- 把你的规矩记下来，每次都让 AI 遵守
- 把做过的事记下来，下次不用从头讲
- 把一个人的经验，变成全队的资产

这件事不性感，但很重要。

AI 编程的早期，大家都在比"模型谁更强""谁生成代码更快"。
但现在，模型能力的差距在缩小。真正拉开差距的，是**你能不能让 AI 稳定地、持续地、按你的标准输出高质量代码**。

模型是发动机。
但只有发动机，没有方向盘、没有刹车、没有导航，车开不快，也开不远。

Trellis 就是那套方向盘+刹车+导航。

它不会让车的马力变大，但它能让你敢把油门踩下去。

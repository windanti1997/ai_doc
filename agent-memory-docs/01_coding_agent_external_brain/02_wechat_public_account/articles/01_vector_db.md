# 01 别再把 Agent 记忆做成向量库了

副标题：人类记忆不是硬盘，Agent Memory 也不该只是 RAG。

建议封面文案：RAG 是查资料，Memory 是长经验。

---

如果一个工程师每天都在重复踩同一个坑，你会觉得他聪明吗？

现在很多 Coding Agent 就处在这个阶段。

它能写代码，能改文件，能跑测试，能读日志，甚至能连续执行几十步任务。但它仍然经常像新人一样：刚踩过的坑，下次还会踩；刚确认过的测试命令，下次又忘；刚理解的项目约束，过几轮就丢。

所以问题不是 Agent 不够会写代码，而是它没有真正的经验系统。

很多人一提 Agent Memory，第一反应就是存聊天记录、存摘要、建向量库、下次检索出来。这当然有用，但它更像资料库，不是完整记忆。

真正值得借鉴的人类记忆，不是“大脑能存多少东西”，而是：大脑如何让过去经历影响未来行动。

## Agent 不是缺上下文，而是缺经验系统

上下文解决的是“这一次任务里你能看到什么”。记忆解决的是“下一次遇到类似问题时你会不会更聪明”。

这两件事很容易被混在一起。

长上下文可以让 Agent 一次性读更多文件、保留更多对话、看到更长的日志。但如果这些东西没有被筛选、验证、沉淀、召回，它们只是在当前窗口里临时存在。

窗口一关，经验就消失了。

人类工程师不会这样。一个工程师踩过一次测试环境坑，下一次会先检查依赖；被 reviewer 指出过一次边界条件，下一次会主动补测试；处理过一次发布事故，之后会形成 checklist。

这才叫长经验。

## RAG 是查资料，Memory 是长经验

RAG 的核心动作是检索：给我一个问题，我从资料库里找相关片段。

Memory 的核心动作是改变行为：因为过去发生过某件事，所以我这一次会换一种做法。

两者不是替代关系。RAG 很适合查 API 文档、历史决策、会议纪要、代码说明。但 Agent Memory 还要回答更多问题：

什么值得记？

这条记忆来自哪里？

它被验证过吗？

它适用于哪个项目、模块、命令、用户或团队？

它现在还成立吗？

它会不会影响下一次行动？

它能不能升级成 skill？

如果没有这些问题，所谓 memory 很容易变成“把历史塞进向量库”。看上去很智能，实际只是更高级的搜索。

## 人类记忆也不是硬盘

人类记忆从来不是完整保存过去。

我们会忘记大量无关细节，会强化高风险事件，会把重复经验变成习惯，会把具体经历抽象成规则，也会因为新证据修正旧判断。

这对 Agent 很有启发。

一次普通命令输出不一定值得存。

一次用户明确纠正，很值得存。

一次失败测试的根因，很值得存。

一个只在旧版本成立的 workaround，应该降权甚至归档。

一个反复出现、步骤稳定、结果可验证的处理流程，应该升级成 skill。

记忆不是仓库，而是生命周期。

## Agent Memory 的第一步不是存储，而是筛选

很多 memory 系统一开始就问：“用什么数据库？”

这个问题太早了。

更应该先问：“哪些经历值得进入长期记忆？”

对 Coding Agent 来说，值得记的通常有几类：

第一类是用户明确纠正。比如“这个项目用 pnpm，不要用 npm”“generated 文件不要手改”“先跑单测再改实现”。这是高价值记忆。

第二类是工具验证结果。比如某个测试命令失败，原因是 Redis 没启动；某个 lint 规则要求 import 排序；某个构建步骤必须先生成 schema。

第三类是高风险模块经验。比如支付、权限、数据迁移、发布流程、安全边界。这些经验即使出现次数不多，也值得加权。

第四类是重复出现的流程。比如调 CI、改 API、做 migration、修 flaky test。如果每次都要重新摸索，就应该沉淀成 workflow 或 skill。

第五类是失败路径。Agent 试过什么方案、为什么失败、怎么验证失败，这些比成功结论更能避免重复踩坑。

## 没有证据的记忆，只是模型的自我感觉

一条好记忆不应该只是：

> payment test 需要 Redis。

它最好能带上证据：

什么时候出现？

哪个命令失败？

错误栈是什么？

哪个文件或服务相关？

如何修复？

最后如何验证？

如果一条记忆没有来源，没有 evidence，没有 last_verified，它就很难被信任。尤其在团队场景里，错误记忆一旦被共享，影响会被放大。

所以 Agent Memory 必须天然带证据链。

这也是为什么 Hook 很重要。PostToolUse 能记录真实命令结果，PreCompact 能保留任务状态，Stop verifier 能防止 Agent 在没验证时假完成。这些都是比事后总结更可靠的记忆写入点。

## 遗忘也是能力

很多人做 memory 时只想“怎么记更多”。

但不会遗忘的 Agent，会越来越难用。

代码库会变，测试命令会变，架构会变，团队规范会变。旧经验如果一直高权重存在，就会污染未来任务。

所以 memory 需要状态：

active：当前有效。

candidate：候选，待验证。

stale：可能过期。

deprecated：已不推荐。

invalidated：被新证据否定。

这不是复杂化，而是工程系统最基本的卫生机制。

## 最高级的记忆会变成技能

Agent Memory 的终点不是 project_memory.md。

project_memory.md 只是在说“知道什么”。真正的成长还要进入“怎么做”。

比如：

Episode：某次 integration test 因 Redis 未启动失败。

Fact：payment integration tests require Redis。

Rule：运行 payment integration tests 前必须检查 Redis。

Skill：payment-test-setup.skill。

Automation：测试前 hook 自动检查 Redis 状态。

这条链路才是 Agent 从经历走向技能的过程。

一个成熟的 Agent 不是记住更多日志，而是形成稳定工作方式：先复现、再定位、再最小修改、再验证、再总结。它会因为长期记忆而改变下一次行动。

## 总结

别再把 Agent 记忆做成向量库了。

向量库可以是 memory 的一个组件，但不是 memory 本身。

RAG 是查资料。

Memory 是长经验。

真正的 Agent Memory 应该包含筛选、证据、加权、巩固、遗忘、召回、技能化。它的目标不是保存过去，而是让 Agent 下一次做得更好。

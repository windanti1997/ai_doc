# 公众号文章写作 SOP

> 本 SOP 用于指导 AI 写作助手产出高质量的公众号技术长文。
> 来源：2026-07-08 Trellis 提效文章写作流程总结。

---

## 一、总览：7 步写作法

```
需求拆解 → 深度调研 → 广度调研 → 竞品对比 → 结构设计 → 正文撰写 → 质量审查
```

每一步都有明确的产出物和检查标准，不达标不进入下一步。

---

## 二、详细流程

### Step 1：需求拆解与任务规划

**目标**：把用户模糊的需求，拆成可执行的任务列表。

**操作**：
1. 启用 anti-overfit-collaboration 协议
2. 识别用户输入类型（执行请求 / 判断请求 / 边界探索 / 开放发散 / 局部批评 / 未澄清问题）
3. 用 TodoWrite 创建 6-8 个任务，每个任务有明确的产出物
4. 高优先级任务排在前面

**产出**：TodoList（含 id、内容、优先级、状态）

**检查**：
- [ ] 任务是否拆解到单步可执行？
- [ ] 是否有明确的完成标准？
- [ ] 优先级排序是否合理？

---

### Step 2：深度调研——官方文档

**目标**：吃透产品/主题的核心概念、架构、价值主张。

**操作**：
1. 优先读官方文档首页、核心概念页、工作原理页、真实场景页
2. 用 WebFetch 精读 3-5 篇核心文档
3. 提取关键信息：
   - 产品定位和核心隐喻（如"辅助轮"）
   - 核心架构和分层（如 spec / task / workspace 三层）
   - 关键术语和定义
   - 官方推荐的使用场景
   - 官方 FAQ 中的常见顾虑

**产出**：核心概念笔记（内化在脑中，不一定要单独写文件）

**检查**：
- [ ] 能不能用一句话说清这个产品是干什么的？
- [ ] 能不能说出 3 个核心设计？
- [ ] 能不能说出官方推荐的 5 个使用场景？

---

### Step 3：广度调研——外部社区

**目标**：找到真实用户的使用体验、推荐语、踩坑记录，让文章有"别人也这么说"的可信度。

**操作**：
1. 用 WebSearch 搜索多渠道：
   - Reddit / Hacker News / X（Twitter）
   - 知乎 / 掘金 / 博客园 / 微信文章
   - 独立博客 / 评测网站
2. 搜索关键词组合：
   - `[产品名] review / experience / 体验 / 评测`
   - `[产品名] vs [竞品名] comparison / 对比`
   - `[产品名] workflow / productivity / 提效`
3. 用 WebFetch 精读 3-5 篇高质量内容
4. 提取可用素材：
   - 真实用户的推荐语（可以 paraphrase）
   - 真实使用场景和案例
   - 常见的批评和踩坑记录
   - 有冲击力的数字或对比

**产出**：素材库（引用源 + 核心观点）

**检查**：
- [ ] 至少有 3 个独立来源？
- [ ] 有正面评价也有负面评价吗？（避免一边倒，显得真实）
- [ ] 有具体的案例或数字吗？（不是空泛评价）

---

### Step 4：竞品对比分析

**目标**：建立对比维度，说清"为什么选这个而不是那个"。

**操作**：
1. 识别用户提到的竞品（如 Superpowers / Spec / Plan）
2. 对每个竞品，搞清楚：
   - 它的核心定位是什么？
   - 它解决了什么问题？
   - 它的优缺点是什么？
   - 它和主角的本质区别是什么？
3. 建立对比维度表格（至少 5 个维度）：
   - 解决的核心问题
   - 上手门槛
   - 系统复杂度
   - 持久化记忆
   - 多平台支持
   - 适合场景
4. 每个竞品都要"先肯定，再指出局限"，不要踩一捧一

**产出**：竞品对比表 + 每个竞品的优缺点分析

**检查**：
- [ ] 竞品分析公平吗？有没有故意贬低？
- [ ] 对比维度有说服力吗？
- [ ] 读者看完能理解"为什么选 Trellis"吗？

---

### Step 5：文章结构设计

**目标**：设计情绪递进的爆款结构，让读者一路读到底。

**公众号爆款结构模板**：

```
1. 【钩子】日常场景 → 制造冲突 → 引发共鸣
   - 用读者熟悉的日常画面开头
   - 点出痛点，制造"对啊我也这样"的共鸣
   - 用一句话升级痛点（"这不是模型不够聪明，是缺了脚手架"）

2. 【痛点放大】把隐性痛点显性化
   - 拆解痛点的具体表现
   - 用"90% 的时间在重新解释"这类对比句强化
   - 让读者觉得"说到我心里了"

3. 【探索过程】我试过的方案，各有各的问题
   - 讲 2-3 个同类方案
   - 每个都先讲优点，再讲为什么不够
   - 让读者跟着一起"试了一圈都不行"，产生期待

4. 【主角登场】答案揭晓，核心价值
   - 用一个好的隐喻（辅助轮 / 方向盘 / 脚手架）
   - 分层讲解核心架构（3 层左右最好记）
   - 每一层都讲清楚：解决了什么问题？怎么解决的？

5. 【真实案例】Good Case / Bad Case
   - 至少 3 个 Good Case，覆盖不同场景
   - 至少 2 个 Bad Case，显得真实可信
   - 每个案例都要有：背景 → 怎么做 → 结果如何

6. 【场景指南】不同情况怎么用
   - 列出 4-6 个典型场景
   - 每个场景说清：适合时机、价值、怎么做
   - 加一张个人 vs 团队对比表

7. 【解除顾虑】FAQ
   - 提前想到读者会问的问题
   - 支持哪些平台？协议有问题吗？会不会变慢？
   - 每个回答都要直接、不回避

8. 【金句收尾】升华主题 + 行动引导
   - 用一个有力的比喻收尾（发动机 vs 方向盘）
   - 回到开头的痛点，形成闭环
   - 自然引导互动（评论区留言）
```

**产出**：文章大纲（标题 + 各节核心观点）

**检查**：
- [ ] 开头有钩子吗？读者会想继续看吗？
- [ ] 情绪是递进的吗？（共鸣 → 焦虑 → 期待 → 理解 → 认可 → 行动）
- [ ] 有真实案例吗？不是空讲道理吧？
- [ ] 结尾有互动引导吗？

---

### Step 6：正文撰写

**目标**：按结构写出完整、流畅、有说服力的文章。

**写作原则**：
1. **用第一人称叙事**："我试过""我踩过坑""我用下来的感受"，有代入感
2. **用具体场景，不用抽象概念**：不说"上下文丢失"，说"第二天开新会话，AI 又问你项目用什么 ORM"
3. **先讲问题，再讲方案**：每个功能点都先讲"以前有什么痛点"，再讲"它怎么解决"
4. **正反都讲**：有 Good Case 也有 Bad Case，显得真实可信
5. **金句密度**：每 300-500 字有一句让人想划线的话
6. **字数控制**：3500-5000 字（公众号长文最佳长度）

**格式规范**：
- 主标题：有冲突/有数字/有反差（"用了三个月，我终于敢让 AI 独立干活了"）
- 副标题：补充核心信息（"从 X 到 Y 再到 Z，我踩过的坑"）
- 小标题：用判断句，不用名词短语（不说"Spec 规范层"，说"你的规矩，AI 每次都遵守"）
- 代码块：适量，不要太多，用来说明具体规则
- 表格：用于对比，清晰易读
- 结尾互动：明确引导评论，最好有小奖品刺激

**产出**：完整的 Markdown 文章文件

**检查**：
- [ ] 标题有吸引力吗？
- [ ] 开头 3 段能抓住人吗？
- [ ] 有金句吗？至少 3-5 句？
- [ ] 案例真实具体吗？
- [ ] 结尾有互动引导吗？
- [ ] 字数在 3500-5000 之间吗？

---

### Step 7：质量审查（反过拟合）

**目标**：用 anti-overfit-reviewer 做最终质量检查。

**操作**：
1. 调用 anti-overfit-reviewer skill
2. 对照审查标准检查：
   - 有没有讨好用户、忽略事实？
   - 有没有把推演说成事实？
   - 有没有虚假的精确数字？
   - 交付物完整可用吗？
   - 有没有关键假设没说明？
3. 如果 FAIL，按要求修改后重新审查

**产出**：审查结论（PASS / PASS WITH RISKS / FAIL）

**检查**：
- [ ] 审查通过了吗？
- [ ] 风险点都清楚了吗？
- [ ] 需要调整的地方都改了吗？

---

## 三、文件输出规范

文章文件统一放在 `trellis-article/` 或对应的项目目录下，命名规则：

```
NN_主题_keyword.md
```

例如：
- `01_trellis_wechat_article.md`
- `02_vector_db_wechat_article.md`

---

## 四、以后怎么触发

**用户说**："写一篇关于 [主题] 的公众号文章"

**我自动执行**：
1. 启用 anti-overfit-collaboration
2. 按本 SOP 的 7 步流程执行
3. 每一步用 TodoWrite 跟踪进度
4. 完成后用 anti-overfit-reviewer 审查
5. 交付最终文章 + 过程总结

**如果用户有额外要求**（比如指定结构、指定案例、指定字数），在 Step 1 需求拆解阶段纳入调整。

---

## 五、质量红线

出现以下情况，文章不能交付：

1. **没有调研就写**：只靠内部知识，不查官方文档，不搜外部社区
2. **一边倒吹捧**：只讲优点，不讲缺点和局限
3. **空泛无物**：全是大道理，没有具体案例和场景
4. **AI 味重**：结构僵化，语言模板化，不像真人写的
5. **虚假精确**：把推演数字说成实测事实
6. **没有互动**：结尾干巴巴，不引导读者评论

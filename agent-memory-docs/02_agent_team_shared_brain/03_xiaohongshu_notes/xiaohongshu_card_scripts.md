# 小红书图文脚本：给 Agent 团队装一个共享大脑

## 账号定位建议

适合定位为：AI 团队协作 / Agent Memory / AI 工程化 / 多 Agent 系统。

第二系列小红书不要讲太多理论，重点放在团队协作痛点：

- Agent 记错了怎么办
- 个人经验能不能共享给团队
- 多个 Agent 会不会互相污染上下文
- 团队规则如何自动提醒

---

## 选题 01：个人记忆不能直接变成团队规则

### 封面标题

个人记忆不能直接变成团队规则

### 卡片结构

1. 单个 Agent 记错，影响一次任务。
2. 团队 Agent 记错，可能影响所有人。
3. 个人偏好 ≠ 项目事实。
4. 项目经验 ≠ 团队规则。
5. 团队规则 ≠ 组织规范。
6. 每条 memory 都要有 scope。
7. Personal / Project / Team / Org / Skill。
8. 个人经验可以成为候选，但不能自动升级。
9. 总结：记忆不是越共享越好，而是要在正确作用域内生效。

---

## 选题 02：Agent 不仅要记住，还要知道凭什么相信

### 封面标题

Agent 不仅要记住，还要知道凭什么相信

### 卡片结构

1. 同一句记忆，来源不同，可信度完全不同。
2. 模型猜测：低可信。
3. 用户说明：中可信。
4. 测试结果：高可信。
5. CI 日志：高可信。
6. 团队 review：更高可信。
7. 每条 memory 都要有 source / evidence / confidence。
8. 没有来源的记忆，不能进入团队规则。
9. 总结：团队记忆必须可追溯。

---

## 选题 03：最高级的记忆，是未来自动提醒

### 封面标题

最高级的记忆，是未来自动提醒

### 卡片结构

1. 很多事故不是不知道规则，而是关键时刻没人想起来。
2. 人类有前瞻记忆：将来遇到 X 时做 Y。
3. Agent 也应该有这种记忆。
4. 改 API → 自动提醒更新 schema。
5. 改 payment → 自动触发安全 checklist。
6. 改 migration → 检查 rollback。
7. 改 auth → 触发权限 review。
8. 记忆不只是内容，也可以是 trigger。
9. 总结：真正有用的团队记忆，会在正确时机出现。

---

## 选题 04：团队不是共享所有知识，而是知道谁知道什么

### 封面标题

团队不是共享所有知识，而是知道谁知道什么

### 卡片结构

1. 真实团队不会让每个人知道所有事。
2. 高效团队知道谁懂支付、谁懂发布、谁能 review。
3. Agent Team 也一样。
4. Planner Agent 需要需求和路线。
5. Coder Agent 需要代码和测试。
6. Reviewer Agent 需要 review 规则。
7. Security Agent 需要安全边界。
8. Memory Routing：按角色给记忆。
9. 总结：团队记忆的关键不是共享，而是路由。

---

## 选题 05：Agent Memory 也需要免疫系统

### 封面标题

Agent Memory 也需要免疫系统

### 卡片结构

1. 错误记忆一旦共享，影响会放大。
2. 错误经验变成 skill，更危险。
3. 团队 memory 不能直接写入。
4. 候选记忆先隔离。
5. 高影响记忆要审批。
6. 敏感信息要脱敏。
7. 过期规则要降权或归档。
8. 错误记忆要能回滚。
9. 总结：共享记忆必须被治理。

## 通用结尾模板

第一代 Agent Memory 解决“我怎么让 Agent 记住我”。

下一代 Agent Memory 要解决：

> 团队如何安全地共享 Agent 的经验？

这才是 Agent 进入真实工程组织后的关键问题。

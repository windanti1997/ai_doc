# Article Contract 09：一个 Agent 踩过的坑，为什么另一个 Agent 还要再踩一遍？

## 文章编号

给 AI 装一个外置大脑 09

## 系列位置

第 8 篇讲记忆分层。本篇在“团队层”继续展开：当一个团队有多个 Agent 或多人共用 Agent 时，经验如何共享，如何避免重复踩坑。

## 标题方向

主标题：一个 Agent 踩过的坑，为什么另一个 Agent 还要再踩一遍？

备选标题：团队记忆不是知识库，是让 Agent 少重复犯错 / Claude Code 修过的坑，Codex 为什么还会再踩？

## 本篇只解决一个问题

跨 Agent、跨会话、跨成员的经验复用。团队记忆不是把所有聊天共享，而是让已验证的项目经验能被需要的 Agent 看到。

## 读者核心情绪

> 这不是刚有人修过吗？怎么换个 Agent 又重来？

## 本篇认知推进

上一篇讲记忆分层。本篇只讲团队层经验共享。下一篇会讲共享记忆会带来污染、冲突、恶意注入，需要免疫系统。

## 必写场景

至少写 3 个：

1. Claude Code 修过一个构建坑，Codex 第二天又踩；
2. 一个同事让 AI 摸清了项目测试命令，另一个 Agent 还是重新问；
3. 代码库里已经有实现约定，但新 Agent 没读到团队经验；
4. 某个坑不是个人偏好，而是项目公共经验，应被团队共享。

## 禁止展开

不讲团队知识库产品大全；不讲权限细节；不讲安全免疫细节；不讲完整多 Agent 架构。本文只讲“经验为什么需要跨 Agent 共享”。

## 推荐开头

从风逆让 Codex 接手一个 Claude Code 昨天刚修过的项目坑开始。结果 Codex 重新踩坑，风逆意识到：这不是某个 Agent 不聪明，而是团队经验没有被共享。

## 叙事路线

场景：不同 Agent 重复踩同一个坑。冲突：每个 Agent 都像第一次来项目。推进：团队不能靠人当传话筒。机制：共享经验必须是已验证、可触发、有适用范围的。落点：团队记忆的价值是减少重复学习成本。

## 结尾任务

落到：一个 Agent 学到的经验，如果对整个项目有效，就不该只留在它自己的会话里。引出第 10 篇：但共享之前，必须先防止坏记忆污染整个团队。

## 封面提示词

封面大字：一个坑 / 两个 Agent 都踩了。封面副标题：团队记忆不是聊天记录共享。画面：轻漫画、浅底、黑红大字、黄色强调条。左边 Agent A 从坑里爬出来贴上“已修复经验”；右边 Agent B 正要再次掉进同一个坑；风逆在中间扶额。黄色便签：“刚修过”“又踩了”“经验没共享”。中央安全区兼容公众号裁剪。

## 跑偏风险

不要写成协同办公工具介绍。必须聚焦 Coding Agent / AI Agent 反复踩项目坑的具体体验。

## 评审重点

普通读者能理解“别让我当 Agent 之间的传话筒”；技术读者能看到团队记忆的边界：不是全量共享，是共享经过验证的项目经验。

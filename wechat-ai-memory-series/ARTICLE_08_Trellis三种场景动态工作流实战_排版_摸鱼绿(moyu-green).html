实战 方法论

上个月做一个用户邀请系统，前后拆了十几个任务，我让 AI 用 Trellis 默认流程跑。

结果呢？

第一个任务写完了，发现接口字段名跟第二个任务对不上。第二个任务做了一半，发现第一个任务的实现有问题，得回去改。改完第一个，第二个又得跟着改。改到第五天，我发现 AI 偷偷加了三个不在需求里的功能——它觉得"用户可能会需要"。

最气人的是，整个项目做完，核心功能是有了，但跟我最开始想要的东西已经不是一个东西了。AI 把它简化成了一个"能用但不好用"的 MVP。

不是 Trellis 不好用，是我用错了

默认 workflow 是给小任务设计的——brainstorm 一下，implement，check，搞定。但大项目不能这么玩。大项目需要路线锁定、需要任务契约、需要并行执行、需要反 MVP 化。

Trellis 不是只有默认配置。它有个隐藏技能叫 trellis-meta，专门用来定制工作流。一个 markdown 文件，改几行，整个工作流就变了。

这篇文章分享我给大需求长期任务定制的完整 workflow，顺带提一下 bugfix 和性能优化的变体。全部基于 Trellis 官方能力，不用改源码，只改 workflow.md 一个文件。

先搞懂：trellis-meta 是什么

THE HIDDEN CUSTOMIZATION TOOL

很多人用了很久 Trellis，都不知道有 trellis-meta 这个东西。

它是 Trellis 自带的一个 skill，作用就一个：帮你理解和修改 Trellis 本身的架构和工作流。


改 workflow

跟它说一声，它告诉你改哪个文件、怎么改。不用猜。


加自定义 agent

想加个专门做代码审查的 agent？跟它说，它告诉你怎么注册。


讲内部架构

想搞清楚 Trellis 内部怎么跑的？它给你讲分层架构，从 workflow 到 hook 到 task system。

最重要的是，它告诉你一个核心事实：Trellis 的工作流全部定义在 .trellis/workflow.md 这一个文件里。

✦ workflow.md 控制四件事

01

Phase 定义 — Plan/Execute/Finish 三个阶段里有哪些步骤

02

Skill Routing 表 — 用户说什么话，自动加载哪个 skill

03

workflow-state 块 — 每轮注入的面包屑提示，告诉 AI 当前状态

04

Task System — task.py 的命令目录

全是纯文本，改完立刻生效。不需要重新安装，不需要改 Python 脚本，不需要 re-init。

大项目定制 workflow，本质就是改这一个文件

PART


大需求长期任务：五层防御

FIVE-LAYER DEFENSE SYSTEM

这是本文的重点。默认 Trellis 跑大项目为什么容易翻车？因为 Plan 阶段太浅了——brainstorm 一下就开干，等于裸奔。

我给大任务加了五层防御，层层递进。

第一层：意图澄清——用第一性原理找到根问题

人跟 AI 说需求，说不全是常态。

你说"做个邀请系统"，心里想的是"微信分享 + 邀请码 + 多级返佣 + 排行榜"，但 AI 听到的就是"邀请系统"四个字。等它做完了，你发现怎么跟你想的不一样——它只做了最基本的邀请功能。

这不是 AI 的锅，是你没说清。但更深一层的问题是：你自己可能也没想清楚。

第一层防御是 anti-overfit-collaboration，但它不只是"问清楚需求"这么简单。它用的是第一性原理拆解：

第一性原理拆解三步法

1. 剥离表象，找到根问题

用户说的"邀请系统"是解决方案，不是问题本身。把手段剥掉，问：用户真正想解决的是什么？是拉新？是裂变？还是提高留存？

2. 还原基本要素

把根问题拆成不可再分的基本要素——谁发起、谁参与、激励是什么、分发路径是什么、成功判据是什么。

3. 从要素重建解法

不基于"邀请系统"这个既有方案，而是从基本要素出发，重新推导可能的解法路径。

你以为你要的是邀请系统，但第一性原理拆完可能发现——你真正要的是"用户自发传播机制"，邀请码只是其中一种实现方式。

怎么拆？用苏格拉底提问。不是"你还有什么要补充的吗"这种客气话，是一针见血的追问：

澄清型："你说的'邀请'具体指什么场景？能不能举一个你遇到的真实例子？"

假设型："你假设邀请了就会来，这个假设成立吗？如果用户邀请了但对方不点呢？"

原因型："为什么你觉得邀请系统是关键？是什么让你得出这个结论？"

视角型："如果站在被邀请的人角度，他为什么要点这个链接？"

推论型："如果按这个方向走，邀请来了之后呢？再下一步呢？"

反思型："我们是不是在讨论真正的问题？有没有可能我们问错了问题？"

一次只问一两个，问到根上。

这一层的产出不是代码，是 prd.md 开头的一段"真实意图 + 根问题"。把模糊的需求钉死在文字上。

别小看这一步。很多项目做到一半跑偏，根源就在这里——最开始就没找对问题。

第二层：路线锁定——防止做着做着方向变了

大项目最容易出现的问题不是做不完，是做着做着方向变了。

今天想"先做基础功能"，明天觉得"这个体验也很重要"，后天又说"这个玩法也得加上"。两周后你发现——怎么跟最开始想的完全不是一个东西了？

我用 route-lock-guardian 做第二层防御。它在项目开始时产出三个文件：


route-lock.md

核心路线是什么，成功标准是什么


non-goals-and-killed-routes.md

明确说清楚什么不做，哪条路已经被杀死了


permission-matrix.md

AI 能自己拍板什么，什么必须问你

做完这一步，项目的边界就定死了。之后每做一个决策，都要对照路线文件——有没有偏离？是不是引入了已经杀死的路线？

"什么不做"比"做什么"更重要

大项目失控，往往不是因为原定功能没做完，是因为加了太多不在计划里的东西

第三层：方案挑刺——苏格拉底提问的反向应用

意图清楚了，方向锁死了，接下来 brainstorm 出具体方案。

但 brainstorm 出来的方案，你懂的——看起来很美，实际跑不通。边界情况没考虑，失败模式没想过，依赖关系理不清。

我加了一步 grill-me——专门挑刺。

挑刺不是乱骂，它本质上是苏格拉底提问的反向应用。第一层用苏格拉底提问是帮用户想清楚，这一层用苏格拉底提问是帮方案找漏洞：

假设型："你假设用户会按这个流程走，如果不呢？"

边界型："数据量上来了，性能扛得住吗？临界点在哪？"

依赖型："这个第三方服务挂了怎么办？有降级方案吗？"

视角型："如果站在攻击者角度，这个方案有什么漏洞？"

推论型："如果按这个方案做下去，半年后会遇到什么问题？"

反思型："我们是不是在解决错误的问题？有没有更简单的办法？"

一个问题一个问题过，每个问题都要回答。答不上来的，回去改方案。

更狠的是自我苏格拉底提问——在方案出街之前，AI 先自己拷问自己一遍：

1. 我回答的是真正的问题，还是我理解的那个容易回答的问题？

2. 我有没有偷偷替换了问题的边界？（把难做的换成好做的）

3. 我的结论建立在什么隐含假设上？这些假设用户同意吗？

4. 有没有可能我和用户都问错了问题？

5. 这个方案的反例是什么？在什么情况下不成立？

七个问题答不上来，说明方案还不够扎实，不能进入下一步。

好的方案不是想出来的，是被骂出来的

第四层：任务契约——完整薄片，不能残缺

这一层最关键，也是我踩过最深的坑。

你有没有过这种经历：AI 说"做完了"，你一看——功能是有了，但最核心的体验都被砍了。动画没了，异常处理没了，边缘情况没考虑。它做了一个"能用"的版本，但不是你想要的版本。

这就是 MVP 化——AI 为了快速完成任务，自动把需求降级成了最小可用版本。

trellis-task-contract 就是干这个的。它产出一份不可逃逸的任务契约，但不止是"写清楚验收标准"这么简单。它有三个核心设计：

01 任务可以小，但不能残

很多人拆任务的方式是"先搭壳，后补逻辑"——第一天写接口定义，第二天写实现，第三天写测试。这叫残缺阶段，每一步都不能独立运行，每一步都制造假进展。

正确的拆法是拆成完整薄片：每个子任务都有完整的输入、处理、输出、接入点、测试和证据。它可能很小，但它是活的——你能跑起来，能测，能验证。

残缺阶段 VS 完整薄片

❌ 残缺阶段

搭接口定义
写业务逻辑
补测试
三个都做完才能跑

→

✅ 完整薄片

用户注册（全）
用户登录（全）
忘记密码（全）
每个做完都能跑

完整薄片的好处是：每个任务完成了就是真的完成了，不会"差不多做完了"然后发现还差一半。

02 测试不是可选的，是契约的一部分

很多任务契约里写"需要有测试"，三个字就完了。但"有测试"和"测试有用"是两回事。

trellis-task-contract 的测试契约写清楚三件事：


单测覆盖什么

核心逻辑、边界条件、异常路径


集成测试覆盖什么

模块交互、调用链、端到端流程


不测什么，为什么

明确测试边界，不是测越多越好

而且还要过测试审查——还是用苏格拉底提问拷问测试方案：

• 这个测试真的验证了逻辑，还是只验证了"函数能跑通"？

• 如果核心逻辑写错了，这个测试能发现吗？还是照样绿？

• 只测单个函数有意义吗？模块拼起来会不会炸？

• 这个测试通过了，能说明功能真的可用吗？

答不上来的，打回去重写，不准进入实现。

03 中间 checkpoint 回填证据

长任务不能等最后才验收。必须设计中间 checkpoint，每个 checkpoint 都要回填证据：

每个 checkpoint 都要回填：

• 改了哪些文件

• 加了哪些测试

• 跑了什么命令

• 结果是什么

• 还有什么没覆盖

每个 checkpoint 都是一个小验收。这样你就不会等到最后才发现——前面全是假进展。

这三个设计合在一起，再加上禁止降级项、反偷懒验收、证据计划，才构成一份完整的任务契约。AI 想蒙混过关？先过了契约这一关再说。

第五层：测试先行——先想清楚"怎么验证"

最后一步，TDD 确认测试边界。

每个任务要测什么？seam 在哪？怎么证明做对了？

不是说先写测试就一定好，是先想清楚"怎么验证"，才能确保做出来的东西是你想要的。如果一个功能不知道怎么测，那它大概率也不知道怎么实现。

TDD 在这里的角色不是"为了测试而测试"，是用测试来定义"做对了"是什么。在写第一行代码之前，先把验收标准想清楚。

怎么想清楚？还是苏格拉底提问——对测试方案自己拷问自己：

• 这个测试真的验证了核心逻辑，还是只验证了"函数能跑通"？

• 如果核心逻辑写错了，这个测试能发现吗？还是照样绿？

• 边界条件、失败路径、异常输入，都覆盖了吗？

• 只测单个函数有意义吗？模块拼起来会不会炸？

• 这个测试通过了，能说明功能真的可用吗？

• 我有没有在测试我想测的东西，还是在测试容易测的东西？

答不上来的，说明测试方案还不够扎实，回去重想。

到这里，Plan 阶段才算完。

你数一下——默认 Trellis 的 Plan 只有 brainstorm 一步，我加到了五层。不是我喜欢复杂，是大项目裸奔的代价太大了。

多花半小时在 Plan 阶段，能省三天的返工

执行阶段：worktree 三批次并行

PARALLEL EXECUTION

方案定了，任务拆了，就到执行了。

大需求的执行跟小任务不一样——很多任务可以同时做。你总不能让 AI 一个一个串行做，那跟自己写有什么区别？

我用的是 git worktree + Codex Goal 的方式，分三个批次：

Batch 1

基础设施
单会话串行

→

Batch 2

并行任务
worktree 多会话

→

Batch 3

集成验证
单会话收尾

Batch 1 基础设施：单会话串行做。因为大家都依赖它，它不做完，后面的都没法开始。

Batch 2 并行任务：每个任务一个 worktree，开一个独立会话，各自跑自己的 Trellis task。每个 worktree 有自己的 git 分支、自己的 active-task 指针、自己的实现。互不干扰，错了就回退各自的分支。

Batch 3 集成：全部合进来，做集成测试和最终验证。

并行执行的关键不是技术，是拆分。每个并行任务必须有独立的测试边界，不能依赖另一个并行任务的产出。Batch 1 做完了，Batch 2 的每个任务拿到的接口都是现成可测的。

⚠️ 踩坑提示

拆得好，并行就是线性加速。拆不好，全是合并冲突，反而更慢。

另外两个场景：顺带一提

BUGFIX + PERFORMANCE

大需求是重头戏，但日常工作里 bugfix 和性能优化也不少。简单说一下这两个场景怎么改。

Bugfix：先诊断，再修

很多人修 bug 的方式是：把报错贴给 AI，AI 改代码，测试通过，搞定。

但过两天同样的 bug 又冒出来了。

为什么？因为 AI 改的是症状，不是根因。

我给 bugfix 场景加了 diagnosing-bugs，强制走科学调试流程：

科学调试六步法（前四步不许改代码）

1. 构建反馈回路——找到一个能让 bug 变红的方法

2. 复现并最小化——用最小的输入复现 bug

3. 生成 3-5 个可证伪假设

4. 逐变量插桩验证——一个一个排除

5. 修复 + 回归测试

6. 清理 + 事后复盘

找到根因之后，先写一个能复现 bug 的失败测试，再修。修完这个测试永远留在测试套件里，防止以后再回来。

最后用 trellis-break-loop 做根因分析——为什么会发生？怎么防止再发生？

一次 bug 是教训，两次同样的 bug 是事故

性能优化：先测，再动

性能优化最容易出两个问题：优化了错误的地方，或者没有 baseline 全靠体感。

我给性能优化场景的改法是：


第一步：先 profiling

前端用 Lighthouse，Android 用 Perfetto + hprof。数据出来了瓶颈在哪一目了然。不要猜。


第二步：先质疑方向

这真的是瓶颈吗？优化收益有多大？值不值得花时间？用 grill-me 挑一挑刺。很多时候挑完发现根本不值得做。


第三步：建 baseline

行为 baseline（测试全绿吗？）+ 性能 baseline（当前数据是多少？）。优化前没有 baseline，优化后你根本不知道有没有效果。


第四步：优化，对比数据

每优化一步跑一次 benchmark。优化了半天提升 5%？那不如不做。

没有 profiling 数据的性能优化，都是瞎优化

你怎么开始：三步上手

说了这么多，你可能想问：我也想定制，从哪开始？

很简单，三步。

第一步

调用 trellis-meta
看当前配置

→

第二步

从最痛的场景
加一个 skill 路由

→

第三步

用起来
边用边迭代

第一步：在 Trellis 项目里跟 AI 说"用 trellis-meta 看看我的 workflow 配置"，它会给你讲清楚当前的 Phase 定义、Skill Routing、状态管理。先搞清楚现状，再谈改。

第二步：不要一上来就改全套三个场景。从你最痛的那个开始——大项目翻车最多就先改长期任务，天天修 bug 就先改 bugfix 的。改什么？不用多，加一个 skill 路由就行。比如 bugfix 场景，在 Skill Routing 表里加一行："用户说 bug/报错 → 加载 diagnosing-bugs"。就这一行，体验立刻不一样。

第三步：改完跑几个项目看看。哪里不顺手，再调。workflow 不是一次成型的，是用出来的。你看我的五层防御，也不是一天想出来的——是翻了好几次车，一层一层加上去的。

CONCLUSION

很多人追求"全自动 AI 编程"，希望 AI 自己理解需求、自己拆任务、自己写代码、自己测试，人什么都不用管。

我觉得这是个伪命题。

AI 能帮你做很多事，但关键决策必须人来做——需求的真实意图是什么？项目的边界在哪？性能优化值不值得做？这些问题 AI 答不了，只有你能答。

好的 workflow 不是把人排除在外，是把人的精力放在真正需要人判断的地方，把机械重复的事交给 AI。

意图澄清需要你拍板，路线锁定需要你签字，方案挑刺需要你回答，UI 验证需要你看一眼——但中间那些拆任务、写代码、跑测试、查规范的事，全让 AI 自己去跑。

这才是 Trellis 真正的价值。

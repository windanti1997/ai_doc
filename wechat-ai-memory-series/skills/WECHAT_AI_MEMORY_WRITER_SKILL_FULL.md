# WeChat AI Memory Writer Skill Pack

## 目标

用于写作、重写和评审公众号文章，尤其是《给 AI / Coding Agent 装一个外置大脑》系列。目标不是写观点正确的文章，而是写读者愿意截图、转发、评论“我也遇到过”的文章。

核心原则：

> 读者有一个说不清的感受，文章替他把它说出来。

## 文章风格

1. 从真实场景进入，不从概念进入；
2. 先让读者共鸣，再轻轻点出机制；
3. 用普通用户能感知的冲突承载技术问题；
4. 不写作者独白式散文；
5. 不把技术内容低幼化，也不堆术语；
6. 每篇必须有独有情绪，不复用上一篇结构。

## 写前必须输出写前卡片

任何公众号文章新写、重写、换视角、换结构、换文体，都必须先输出写前卡片：

```text
【本篇标题】
【本篇读者是谁】
【本篇独有情绪】
【读者内心台词】
【本篇认知推进】
【本篇叙事形状】
【本篇主场景】
【本篇主比喻】
【情绪递进】
【禁用内容】
【失败判定】
```

最重要的是“读者内心台词”。它决定文章是不是在替读者说话。

## 标题严格规则

标题必须先负责点击，再负责概括。

### 标题无上下文测试

标题必须脱离正文也能让读者大概知道文章在讲什么。标题至少交代三项中的两项：

1. 主体：AI / ChatGPT / Agent / 记忆 / 记住；
2. 冲突：记错、乱记、误会、越记越烦、把人记窄；
3. 具体问题：临时要求变长期偏好、旧话被乱套、一次要求被当成人设。

红线：标题里出现“这个”“那次”“它”但没有上下文；标题只像一句吐槽，不知道对象是谁；标题需要读完文章才懂；标题只有情绪没有信息；标题没有 AI 相关识别度；标题只是正文金句。

反例：`别把这个当成我的固定偏好啊`。它可以做文中金句、封面副标题或互动句，但不能做主标题。

主标题必须同时通过：人话测试、无上下文测试、点击测试、系列识别测试。

## 正文结构规则

开头 150～200 字内必须出现：具体人物、具体任务、具体冲突、读者能立刻理解的反应。

中段要推进冲突，而不是罗列案例。推荐节奏：第一次看起来挺好，第二次开始不对劲，第三次发现不是偶然，第四次意识到机制问题，第五次落到读者自己的经历。

观点必须从场景里长出来。少用“本质上”“核心是”“真正的问题是”。这些句式可以出现，但不能替代现场。

结尾不要作文总结，不要“欢迎评论”式运营引导。结尾应自然落到一句读者能带走的人话，但不能解释这句话为什么像人话。

## 排版规则

普通叙述 2～4 句一段。情绪转折、金句、吐槽、对话可以单独成行。连续场景不要每句都断。连续 5 个单句段判不通过。

Doocs / Markdown 规则：不用代码块承载正文金句；不要强行 01/02/03 编号；编号只适合教程、清单、方法论，不适合连续情绪叙事；使用自然小标题、引用、加粗和分割线。

## 第三人称规则

第三人称人物默认使用“风逆”。第三人称不是把“你”替换成“风逆”，而是要有连续处境、动作和心理变化。推荐混合型：风逆故事开头 → 中段转读者共鸣 → 结尾落读者共同心声。中段转向读者后，结尾不要突然回到风逆。

## 反 AI 味规则

禁止大量排比、对称句、机械金句、作者升华、案例库堆叠、过度工整句式、逐句断行、人物只是观点传声筒、段落像“现象-本质-总结”。

连续三句以上同结构默认有 AI 味风险。抽象概念词出现后，必须立刻落回具体场景、动作或人话。

## 禁止自证规则

正文禁止出现：这是一句人话；这句话更像真实用户会说；这句话比某个概念更自然；读者会想说这句话；这就是一个反常识；这个场景能让读者共鸣。

如果一句话需要作者解释它为什么自然、为什么共鸣、为什么像人话，说明它还不够自然。必须让场景自己证明。

## 禁止金句三连

同一句核心表达不能连续以“小标题 → 引用 → 解释”的形式重复出现。

反例：

```text
## 上次是上次，这次不是那个场景
很多被 AI 乱记的瞬间，最后都会落到一句人话上：
> 上次是上次，这次不是那个场景。
这句话比“临时要求不等于长期偏好”更像真实人在用 AI 时会冒出来的反应。
```

判定：必须改。问题是重复、自证、讨好人话评审。

## 固定评审团

必须执行 15 个评审：标题评审、爆款编辑、普通读者、情绪递进、反 AI 味、系列主编、公众号排版、互动转发、Doocs / Markdown 排版、叙事人称与角色代入、人话校验、配图评审、文章整体读者视角、最终主编、反评审讨好。

所有 Subagent 的输出状态只允许：通过 / 必须改 / 不适用。禁止：可接受、基本通过、问题不大、可选优化、整体还行。

每个 Subagent 必须输出：

```text
【结论】通过 / 必须改 / 不适用
【最可能失败点】真实读者最可能在哪出戏
【红线检查】是否触发硬红线
【证据】引用正文具体句子或位置
【修改方向】如果必须改，给出可执行修改
```

## 反评审讨好 Subagent

目标：检查文章是否在迎合评审表，而不是服务读者阅读。

红线：正文出现“这句话更像人话”“这就是冲突”等自证式表达；小标题、引用、解释三连重复同一句核心表达；每段都像逐项满足某个 Subagent；为了通过标题/互动/人话评审，把本该出现在评审里的判断写进正文；评审只引用规则关键词，不引用正文具体失败证据。

硬门槛：必须通过。只要文章有明显迎合评分表痕迹，不得交付。

## 读者连续阅读模拟

最终主编和文章整体读者视角 Subagent 必须模拟普通读者从标题、第一屏、中段、结尾连续阅读的心理变化，而不是逐条打分。

必须回答：读者看到标题是否知道大概讲 AI 的什么问题；第一屏是否有停留理由；中段是否还想继续读；结尾有没有自然落下；哪句话可能让读者转发或评论，是否真的像人话。

## 封面图规则

本系列公众号封面默认：轻漫画插画风、浅色背景、黑红大字标题、黄色强调条、黄色便签、红色警示贴纸。只生成一张图，但必须同时兼容 2.35:1 消息列表裁剪和 1:1 转发卡片裁剪。主标题、主人物脸、AI、核心冲突、副标题必须集中在中央安全区。四周边缘只放辅助元素。

## 归档与 source-of-truth

归档目录只能保存最终原文。禁止摘要、压缩、润色、重写、补写。如果最终原文不可用，必须标记 `NEEDS_SOURCE_BACKFILL`，不能假装完成。
